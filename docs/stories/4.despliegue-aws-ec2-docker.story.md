# Historia #4: Despliegue de Aplicación BTG Pactual V2 en AWS EC2 con Docker

**Estado:** Lista para Revisión

---

## Historia de Usuario

**Como** desarrollador del equipo BTG Pactual V2,  
**Quiero** desplegar la aplicación en una instancia AWS EC2 usando Docker con externalización de secretos y conexión a AWS DynamoDB real,  
**Para** demostrar el funcionamiento completo del sistema en un entorno cloud durante una prueba técnica

## Descripción

La aplicación BTG Pactual V2 (Spring Boot 4.0.3, Java 24, arquitectura hexagonal + CQRS) actualmente funciona exclusivamente en entorno local con DynamoDB Local en Docker. Se requiere crear toda la infraestructura de despliegue necesaria para ejecutar la app en una instancia AWS EC2, contenerizándola con Docker multi-stage, externalizando los secretos sensibles (JWT, AES-256, credenciales DynamoDB) a variables de entorno, y conectándose al servicio AWS DynamoDB real. El despliegue es manual (sin CI/CD) y orientado a demostración en prueba técnica, por lo que se usa HTTP plano y IP pública directa de EC2.

---

## Criterios de Aceptación

### Escenario 1: Build de imagen Docker exitoso

- **Dado** que el Dockerfile multi-stage está configurado con Gradle JDK 24 (build) y JRE 24 (runtime)
- **Cuando** se ejecuta `docker build -t btg-pactual-v2 .` desde la raíz del proyecto
- **Entonces** la imagen se construye sin errores, el JAR se genera correctamente en la etapa de build y la imagen final de runtime no contiene el código fuente ni las herramientas de compilación

### Escenario 2: Externalización completa de secretos

- **Dado** que se ha creado el archivo `.env.ec2` a partir del template `.env.ec2.template` con valores reales para `JWT_SECRET`, `AES_KEY`, `DYNAMODB_ENDPOINT`, `DYNAMODB_REGION`, `DYNAMODB_ACCESS_KEY` y `DYNAMODB_SECRET_KEY`
- **Cuando** la aplicación arranca con el perfil `ec2` (`spring.profiles.active=ec2`)
- **Entonces** todos los secretos se leen exclusivamente de variables de entorno, no hay secretos hardcodeados en la imagen Docker ni en el código fuente desplegado, y `application-ec2.properties` utiliza interpolación `${VARIABLE}` para cada valor sensible

### Escenario 3: App levanta correctamente en EC2 con DynamoDB real

- **Dado** que la instancia EC2 está corriendo con Docker instalado y el archivo `.env.ec2` contiene credenciales válidas de AWS (access key/secret key con permisos DynamoDB)
- **Cuando** se ejecuta `docker-compose -f docker-compose.ec2.yml up -d`
- **Entonces** el contenedor de la app arranca exitosamente, se conecta al servicio AWS DynamoDB (endpoint regional, no localhost), y la API responde correctamente en `http://<IP_PUBLICA_EC2>:8080/swagger-ui.html`

### Escenario 4: Security Group configurado correctamente

- **Dado** que se ha creado un Security Group en AWS con las reglas documentadas
- **Cuando** un usuario externo intenta acceder a la app
- **Entonces** el puerto 8080 (HTTP) está accesible desde cualquier IP para la API, el puerto 22 (SSH) está accesible para administración, y todos los demás puertos están bloqueados

### Escenario 5: Script de despliegue ejecuta el flujo completo

- **Dado** que se tiene una instancia EC2 con Amazon Linux 2 / Ubuntu y acceso SSH
- **Cuando** se ejecuta el script `deploy-ec2.sh`
- **Entonces** el script instala Docker y Docker Compose (si no están instalados), clona el repositorio, construye la imagen Docker, y levanta la aplicación con docker-compose usando el perfil EC2

## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** Configuración Externalizada por Perfiles Spring + Containerización Multi-Stage

**Justificación:** El sistema ya utiliza el mecanismo de perfiles de Spring Boot y la inyección de propiedades vía `@Value`. La externalización de secretos se logra usando un perfil `ec2` que sobrescribe values con interpolación `${VARIABLE}` — las 4 clases Java que consumen configuración (`JwtProveedor`, `ConfiguracionDynamoDb`, `AdaptadorEncriptacionAes`, `LoginManejador`) NO requieren cambios de código. La containerización multi-stage aprovecha el toolchain Gradle existente para build reproducible sin depender del entorno host. Alternativa descartada: IAM Role para credenciales DynamoDB (más seguro pero innecesario para prueba técnica — se usarán access key/secret key vía variables de entorno).

**Componentes Afectados:**

- **`application-ec2.properties` (Nuevo):** Perfil Spring para EC2 que sobrescribe `jwt.secret`, `encriptacion.clave-aes`, `dynamodb.endpoint`, `dynamodb.region`, `dynamodb.access-key`, `dynamodb.secret-key` con interpolación `${VARIABLE}`. Hereda el resto de `application.properties`
- **`Dockerfile` (Nuevo):** Multi-stage: etapa 1 `gradle:jdk24` compila JAR con `./gradlew bootJar`; etapa 2 `eclipse-temurin:24-jre` copia solo el JAR para runtime minimal. Expose 8080, activa perfil ec2
- **`docker-compose.ec2.yml` (Nuevo):** Define solo el servicio `app` (sin DynamoDB Local). Lee variables desde `.env.ec2`, mapea puerto 8080:8080, activa `SPRING_PROFILES_ACTIVE=ec2`
- **`.env.ec2.template` (Nuevo):** Template con placeholders para todos los secretos. El operador lo copia a `.env.ec2` y completa valores reales
- **`deploy-ec2.sh` (Nuevo):** Automatiza: instalación Docker/Docker Compose (si no existe), clonación de repo, build de imagen, levantamiento con docker-compose
- **`.gitignore` (Modificación — Menor):** Agregar `.env.ec2` y `*.env` para evitar commit de secretos reales
- **`ConfiguracionDynamoDb.java` (Sin cambios — Verificado):** Ya consume properties vía `@Value`. El `endpointOverride(URI.create(endpoint))` funciona tanto para endpoint local como regional
- **`JwtProveedor.java` (Sin cambios — Verificado):** Ya consume `jwt.secret` y `jwt.expiration-ms` vía `@Value`
- **`AdaptadorEncriptacionAes.java` (Sin cambios — Verificado):** Ya consume `encriptacion.clave-aes` vía `@Value`
- **`LoginManejador.java` (Sin cambios — Verificado):** Ya consume `jwt.max-failed-attempts` vía `@Value`

### Validación de Impacto

**Código fuente verificado:** Se leyeron los 4 archivos Java que consumen configuración sensible (`JwtProveedor`, `ConfiguracionDynamoDb`, `AdaptadorEncriptacionAes`, `LoginManejador`). Todos usan `@Value("${propiedad}")` en sus constructores, lo cual permite que Spring Boot resuelva las propiedades del perfil `ec2` que a su vez interpolan variables de entorno `${VARIABLE}`. **NINGÚN archivo Java requiere modificación** — la externalización es 100% transparente.

**Punto arquitectónico clave:** `ConfiguracionDynamoDb.endpointOverride(URI.create(endpoint))` funciona tanto para `http://localhost:8000` (DynamoDB Local) como para `https://dynamodb.us-east-1.amazonaws.com` (DynamoDB real). No se requiere lógica condicional.

**Riesgos identificados:**

| Credenciales AWS sin permisos DynamoDB | Media | Alto | Documentar permisos mínimos en `.env.ec2.template` |
| Security Group mal configurado | Baja | Medio | Documentación explícita de reglas requeridas |
