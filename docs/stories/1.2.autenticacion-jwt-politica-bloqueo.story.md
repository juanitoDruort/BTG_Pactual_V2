# Historia #1.2: Definición del proceso de autenticación con JWT y política de bloqueo de cuentas

**Estado:** Borrador (PO)

---

## Historia de Usuario

**Como** administrador de la plataforma BTG Pactual,  
**Quiero** contar con una definición conceptual completa del proceso de autenticación mediante JWT y la política de bloqueo de cuentas,  
**Para** que el equipo de desarrollo tenga una especificación clara que guíe la implementación de un mecanismo de autenticación seguro con gestión de sesiones stateless y protección contra accesos no autorizados.

## Descripción

Esta historia define conceptualmente el flujo de autenticación de la plataforma de gestión de fondos de inversión de BTG Pactual. El sistema debe implementar autenticación basada en JWT (JSON Web Token) stateless con tokens de expiración configurable (default 5 minutos). El endpoint de login recibe credenciales (email y contraseña), valida al usuario y retorna un JWT que el cliente utiliza para consumir todos los demás endpoints protegidos enviándolo en la cabecera `Authorization: Bearer {token}`. Incluye política de bloqueo tras intentos fallidos y proceso de desbloqueo por administrador.

---

## Criterios de Aceptación

### Escenario 1: Login exitoso — credenciales válidas retornan JWT

- **Dado** que un usuario registrado con cuenta activa (no bloqueada) accede al endpoint público de login
- **Cuando** envía su email y contraseña correctos
- **Entonces** el sistema valida las credenciales, resetea el contador de intentos fallidos a 0, genera un JWT stateless con claims (userId, rol, fecha de expiración) y lo retorna en el cuerpo de la respuesta con código HTTP 200. Este token es el que el usuario debe enviar en la cabecera `Authorization: Bearer {token}` para consumir todos los endpoints protegidos

### Escenario 2: Login fallido — credenciales inválidas

- **Dado** que un usuario envía credenciales al endpoint de login
- **Cuando** el email no existe o la contraseña es incorrecta
- **Entonces** el sistema incrementa el contador de intentos fallidos del usuario, retorna un error genérico "Credenciales inválidas" (sin revelar si el problema es el email o la contraseña) con código HTTP 401 (Unauthorized)

### Escenario 3: Bloqueo automático tras intentos fallidos

- **Dado** que un usuario ha acumulado 2 intentos fallidos consecutivos de login
- **Cuando** falla un tercer intento de autenticación (configurable, default 3)
- **Entonces** la cuenta se bloquea automáticamente, se retorna un mensaje "Cuenta bloqueada. Contacte al administrador para desbloquearla" con código HTTP 403 (Forbidden), y todos los intentos posteriores de login con esas credenciales retornan el mismo mensaje de bloqueo sin incrementar el contador

### Escenario 4: Acceso a endpoint protegido con token válido

- **Dado** que un usuario se autenticó exitosamente y obtuvo un JWT
- **Cuando** realiza una petición a cualquier endpoint protegido enviando el token en la cabecera `Authorization: Bearer {token}` y el token no ha expirado
- **Entonces** el sistema valida el token, extrae userId y rol, y permite que la petición continúe al proceso de autorización (HU 1.3)

### Escenario 5: Acceso a endpoint protegido con token expirado

- **Dado** que un usuario tiene un JWT cuyo tiempo de expiración (5 minutos por defecto) ya venció
- **Cuando** intenta acceder a un endpoint protegido con dicho token
- **Entonces** el sistema rechaza la petición con mensaje "Token expirado. Inicie sesión nuevamente" con código HTTP 401 (Unauthorized)

### Escenario 6: Acceso a endpoint protegido sin token o token malformado

- **Dado** que se recibe una petición a un endpoint protegido
- **Cuando** la cabecera `Authorization` está ausente, vacía, o el token no es un JWT válido
- **Entonces** el sistema rechaza la petición con mensaje "Token de autenticación requerido" o "Token inválido" con código HTTP 401 (Unauthorized)

### Escenario 7: Desbloqueo de cuenta por administrador

- **Dado** que un administrador autenticado identifica una cuenta bloqueada
- **Cuando** ejecuta la acción de desbloqueo para dicho usuario
- **Entonces** el sistema desbloquea la cuenta, resetea el contador de intentos fallidos a 0, y el usuario puede volver a intentar login normalmente

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Cualquier usuario registrado (Cliente o Administrador) que necesita autenticarse para acceder a la plataforma
- **Permisos requeridos:** Ninguno para login (endpoint público). Rol Administrador para desbloquear cuentas
- **Valor de negocio:** Garantizar que solo usuarios legítimos accedan a la plataforma de gestión de fondos, protegiendo datos financieros sensibles de los clientes. El token JWT permite un modelo stateless escalable

### Reglas de Negocio

- Autenticación mediante JWT stateless (no sesiones en servidor)
- El endpoint de login es público — uno de los dos únicos endpoints sin autenticación (junto con registro)
- Credenciales de login: email + contraseña
- El login retorna un JWT en el cuerpo de la respuesta
- El token JWT incluye claims: userId, rol (CLIENTE/ADMINISTRADOR), fecha de expiración
- Expiración del token: 5 minutos por defecto (valor configurable)
- El token se envía en cabecera `Authorization: Bearer {token}` en cada petición a endpoints protegidos
- Bloqueo automático tras 3 intentos fallidos consecutivos (valor configurable)
- El contador de intentos fallidos se resetea tras un login exitoso
- Solo un administrador puede desbloquear una cuenta bloqueada
- NO se implementa autenticación de dos factores (2FA) — decisión explícita del stakeholder
- Todo el proceso debe incluir manejo adecuado de excepciones con mensajes claros

### Estructura del Token JWT

| Claim | Descripción |
|---|---|
| sub | userId del usuario autenticado |
| rol | Rol del usuario (CLIENTE o ADMINISTRADOR) |
| iat | Fecha/hora de emisión del token |
| exp | Fecha/hora de expiración (iat + 5 minutos configurable) |

### Catálogo de Excepciones de Autenticación

| Caso | Código HTTP | Mensaje |
|---|---|---|
| Credenciales inválidas | 401 Unauthorized | "Credenciales inválidas" |
| Cuenta bloqueada | 403 Forbidden | "Cuenta bloqueada. Contacte al administrador" |
| Token expirado | 401 Unauthorized | "Token expirado. Inicie sesión nuevamente" |
| Token malformado | 401 Unauthorized | "Token inválido" |
| Token ausente | 401 Unauthorized | "Token de autenticación requerido" |

### Sistemas Externos

No aplica — la autenticación es completamente interna usando JWT firmado localmente.
