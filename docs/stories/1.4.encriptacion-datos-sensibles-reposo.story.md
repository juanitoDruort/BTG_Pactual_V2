# Historia #1.4: Definición de la estrategia de encriptación de datos sensibles en reposo

**Estado:** Analizado (Arquitecto)

---

## Historia de Usuario

**Como** administrador de la plataforma BTG Pactual,  
**Quiero** contar con una definición conceptual completa de la estrategia de encriptación de datos sensibles almacenados en el sistema,  
**Para** que el equipo de desarrollo tenga una especificación clara que guíe la implementación de mecanismos de protección de contraseñas y saldos de clientes, garantizando la seguridad de la información financiera en reposo.

## Descripción

Esta historia define conceptualmente la estrategia de encriptación de datos sensibles en reposo para la plataforma de gestión de fondos de inversión de BTG Pactual. Se identifican dos tipos de datos sensibles que requieren protección: contraseñas de usuario (hashing unidireccional) y saldos de clientes (encriptación reversible). Se especifican los algoritmos recomendados, la integración con la arquitectura hexagonal existente, consideraciones de rendimiento, y recomendaciones para la migración futura de persistencia en memoria a base de datos real.

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Equipo de desarrollo y arquitectura que implementará la estrategia de encriptación
- **Permisos requeridos:** No aplica (documento conceptual)
- **Valor de negocio:** Proteger la información financiera sensible de los clientes (saldos y contraseñas) contra accesos no autorizados a nivel de almacenamiento, asegurando que incluso si se accede directamente a los datos persistidos, la información sensible no sea legible

### Reglas de Negocio

- **Datos sensibles identificados:** Contraseñas de usuario y saldos de clientes
- **Contraseñas:** Protección mediante hashing unidireccional — no deben poder recuperarse jamás
- **Saldos:** Protección mediante encriptación reversible — deben poder desencriptarse para operaciones de negocio (cálculos, visualización)
- No aplican requisitos regulatorios específicos (Superfinanciera, PCI DSS, habeas data) según el stakeholder
- La estrategia debe ser compatible con la arquitectura hexagonal existente
- La estrategia debe considerar la migración futura de ConcurrentHashMap a base de datos real
- Todo el proceso debe incluir manejo adecuado de excepciones (fallo al encriptar, fallo al desencriptar, clave inválida, etc.)

### Estrategia de Protección por Tipo de Dato

| Dato Sensible | Estrategia | Algoritmo Recomendado | Justificación |
|---|---|---|---|
| Contraseñas | Hashing unidireccional | BCrypt o Argon2 | Las contraseñas NO necesitan recuperarse. Solo se compara el hash almacenado con el hash de la contraseña ingresada. Hashing unidireccional previene que un atacante recupere la contraseña original |
| Saldos | Encriptación simétrica reversible | AES-256 | Los saldos NECESITAN desencriptarse para operaciones de negocio (descontar saldo, mostrar saldo al cliente). Encriptación simétrica con clave única del sistema |

### Integración con Arquitectura Hexagonal

```
Dominio (domain/)
├── port/out/PuertoEncriptacion.java  ← Puerto/interfaz que define operaciones de encriptación
│   ├── hashear(texto) → textoHasheado
│   ├── verificarHash(texto, hashAlmacenado) → boolean
│   ├── encriptar(dato) → datoEncriptado
│   └── desencriptar(datoEncriptado) → dato

Infraestructura (infrastructure/)
├── adapter/out/security/
│   └── AdaptadorEncriptacion.java  ← Implementación con librerías específicas (BCrypt, AES)
```

- El dominio define la interfaz (QUÉ necesita), infraestructura la implementa (CÓMO lo hace)
- El dominio permanece libre de dependencias de librerías de encriptación
- Si cambia la librería o algoritmo, solo se modifica el adaptador en infraestructura

### Catálogo de Excepciones de Encriptación

| Caso | Manejo Esperado |
|---|---|
| Fallo al hashear contraseña durante registro | Error 500 — "Error interno al procesar registro" (no revelar detalles de encriptación) |
| Fallo al verificar hash durante login | Error 500 — "Error interno de autenticación" |
| Fallo al encriptar saldo | Error 500 — "Error interno al procesar operación" |
| Fallo al desencriptar saldo | Error 500 — "Error interno al consultar datos" |
| Clave de encriptación no configurada | Error de arranque — la aplicación no debe iniciar sin clave configurada |

### Recomendación Complementaria: HTTPS/TLS

Aunque el stakeholder no solicitó explícitamente protección de datos en tránsito, se recomienda formalmente implementar HTTPS/TLS en todas las comunicaciones de la plataforma como práctica de seguridad fundamental, dado que se transmiten credenciales y datos financieros en cada petición.

### Sistemas Externos

No aplica — la encriptación es un proceso completamente interno.

## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** Puerto de Encriptación en Dominio + Adaptador Transparente en Infraestructura — sobre Arquitectura Hexagonal existente

**Justificación:** La historia define dos estrategias de protección fundamentalmente distintas. (1) **Hashing de contraseñas (unidireccional):** Ya resuelta en HU 1.1 mediante `PasswordEncoder` (BCrypt) inyectado en `RegistroManejador`, y verificada en HU 1.2 mediante `LoginManejador`. No requiere un puerto de dominio adicional. (2) **Encriptación de saldos (reversible, AES-256):** Requiere un puerto de dominio (`PuertoEncriptacion`) con implementación en infraestructura (`AdaptadorEncriptacionAes`). La encriptación/desencriptación se aplica de forma transparente en la capa de persistencia (adaptadores de repositorio): al guardar un cliente se encripta el saldo, al leerlo se desencripta. Esto preserva el modelo de dominio intacto (`BigDecimal saldo`) y hace la encriptación invisible para `ServicioSuscripcion`, `Cliente.descontarSaldo()` y toda lógica de negocio.

**Componentes Afectados:**

- **PuertoEncriptacion.java (Nuevo):** Interfaz en `domain/port/out/` que define operaciones de encriptación reversible. Métodos: `String encriptar(String dato)` y `String desencriptar(String datoEncriptado)`. Opera con `String` para ser agnóstica del tipo de dato — la conversión `BigDecimal ↔ String` la hace el consumidor. El dominio define QUÉ necesita sin depender de librerías de criptografía.
  - NOTA: No incluye métodos de hashing (`hashear`, `verificarHash`) porque el hashing de contraseñas ya se resuelve con `PasswordEncoder` de Spring Security en HU 1.1/1.2.

- **AdaptadorEncriptacionAes.java (Nuevo):** `@Component` en `infrastructure/adapter/out/security/` que implementa `PuertoEncriptacion`. Usa `javax.crypto.Cipher` con AES-256/GCM/NoPadding (GCM proporciona autenticación + encriptación). Lee la clave de `application.properties` (`encriptacion.clave-aes`). Genera IV aleatorio por operación de encriptación. El resultado incluye `IV + ciphertext` codificado en Base64. Valida al arranque que la clave esté configurada y tenga 256 bits.

- **AdaptadorClienteEnMemoria.java (Modificación — Menor):** Inyectar `PuertoEncriptacion`. En `guardar()`: antes de almacenar, encripta el saldo del cliente. En `buscarPorId()` y `buscarPorEmail()`: después de leer, desencripta el saldo. Los datos precargados en `cargarDatosIniciales()` se almacenan encriptados. La lógica de negocio recibe siempre `BigDecimal` en texto plano.
  - Nivel de cambio: Menor (agregar inyección + wrap de encriptación en read/write)

- **application.properties (Modificación — Menor):** Agregar propiedad `encriptacion.clave-aes` con clave AES-256 codificada en Base64 (32 bytes = 256 bits). En desarrollo: valor hardcodeado. En producción: variable de entorno o vault de secretos. La aplicación NO debe arrancar si la clave no está configurada.
  - Nivel de cambio: Menor

- **build.gradle (Sin cambios):** No requiere dependencias adicionales. `javax.crypto` (AES, Cipher, SecretKeySpec) viene incluido en el JDK estándar de Java 24.

- **ManejadorExcepcionesGlobal.java (Modificación — Menor):** Los fallos de encriptación/desencriptación deben mapearse a HTTP 500 con mensaje genérico ("Error interno al procesar operación") sin revelar detalles de criptografía. Capturar una nueva `ExcepcionEncriptacion` que extienda `RuntimeException` (no `ExcepcionDominio`, ya que los fallos criptográficos son de infraestructura, no de negocio).
  - Nivel de cambio: Menor

**Hitos de Implementación:**

1. **PuertoEncriptacion.java** — Crear interfaz en `domain/port/out/` con operaciones `encriptar(String)` y `desencriptar(String)`
   - Dependencias: Ninguna
2. **application.properties — clave AES** — Agregar propiedad `encriptacion.clave-aes` con clave Base64 de 256 bits
   - Dependencias: Ninguna (paralelo con hito 1)
3. **AdaptadorEncriptacionAes.java** — Implementación AES-256-GCM con IV aleatorio, lectura de clave de properties, validación de arranque
   - Dependencias: PuertoEncriptacion, application.properties con clave
4. **AdaptadorClienteEnMemoria.java — encriptación transparente** — Inyectar PuertoEncriptacion, encriptar saldo al guardar, desencriptar al leer
   - Dependencias: AdaptadorEncriptacionAes funcional
5. **ManejadorExcepcionesGlobal — ExcepcionEncriptacion** — Handler para errores de criptografía → HTTP 500 genérico
   - Dependencias: Definir ExcepcionEncriptacion

### Validación de Impacto

Se verificó el código fuente real de todos los componentes involucrados:

- **Cliente.java:** Actualmente tiene `BigDecimal saldo` como campo plano sin ninguna encriptación. Los métodos `descontarSaldo()` y `getSaldo()` operan con BigDecimal directamente. No se modifica — la encriptación es transparente en los adaptadores de persistencia.
- **ServicioSuscripcion.java:** Opera con `cliente.descontarSaldo(monto, nombreFondo)` y `repositorioCliente.guardar(cliente)`. No tiene conocimiento de encriptación — recibe BigDecimal plano del adaptador y lo devuelve plano al guardar. El adaptador encripta/desencripta de forma transparente.
- **AdaptadorClienteEnMemoria.java:** Almacena clientes en `ConcurrentHashMap` con constructor `("cliente-1", "Juan Rodriguez", new BigDecimal("500000"))`. Punto exacto donde se aplicará la encriptación del saldo antes de almacenar y desencriptación al leer.
- **build.gradle:** No tiene dependencias de criptografía. No las necesita — `javax.crypto` está en el JDK.
- **application.properties:** Solo tiene configuración base. Se le agrega la clave AES.
- **ManejadorExcepcionesGlobal:** Maneja ExcepcionDominio → 422, validaciones → 400. HU 1.1 agrega 409, HU 1.2 agrega 401/403. HU 1.4 agrega 500 para fallos de encriptación.
- **Tests (ControladorFondoE2ETest):** Usan AdaptadorClienteEnMemoria con datos precargados. La encriptación transparente no rompe los tests si el ciclo encriptar/desencriptar es correcto — el test ve BigDecimal plano.

### Notas Técnicas (Si aplica)

- **Separación clara hashing vs. encriptación:** El hashing de contraseñas (BCrypt) NO pasa por `PuertoEncriptacion`. Ya está resuelto en HU 1.1 con `PasswordEncoder` inyectado en `RegistroManejador`. `PuertoEncriptacion` se limita exclusivamente a encriptación reversible (saldos). Esto evita un puerto "dios" con múltiples responsabilidades.
- **AES-256-GCM vs. AES-256-CBC:** Se recomienda GCM porque proporciona encriptación autenticada (AEAD) — detecta manipulación del ciphertext además de proteger confidencialidad. CBC requeriría HMAC adicional para verificar integridad.
- **IV (Initialization Vector) aleatorio por operación:** Cada llamada a `encriptar()` genera un IV de 12 bytes con `SecureRandom`. El resultado almacenado es `Base64(IV || ciphertext || tag)`. `desencriptar()` extrae los primeros 12 bytes como IV y el resto como ciphertext+tag. Esto asegura que el mismo saldo encriptado dos veces produce resultados diferentes.
- **Clave AES en producción:** La propiedad `encriptacion.clave-aes` es para desarrollo. En producción: usar variable de entorno, Azure Key Vault, o mecanismo equivalente. La clave NUNCA debe estar en control de versiones para entornos productivos.
- **Migración a Cosmos DB:** Cuando se implemente HU 3 (modelo de datos Cosmos DB), el `AdaptadorClienteCosmosDb` inyectará el mismo `PuertoEncriptacion` y aplicará el mismo patrón de encriptación transparente. Alternativamente, Cosmos DB ofrece encriptación en reposo nativa — evaluar si es preferible usar la encriptación del proveedor.
- **HTTPS/TLS (recomendación complementaria):** Aunque no es alcance de esta HU, se recomienda formalmente HTTPS/TLS para protección de datos en tránsito. Spring Boot soporta configuración nativa con certificados en `application.properties`.
