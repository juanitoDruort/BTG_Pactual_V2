# Historia #1.3: Definición del proceso de autorización por roles y aislamiento de datos

**Estado:** Lista para Revisión

---

## Historia de Usuario

**Como** administrador de la plataforma BTG Pactual,  
**Quiero** contar con una definición conceptual completa del proceso de autorización basada en roles y las reglas de aislamiento de datos por usuario,  
**Para** que el equipo de desarrollo tenga una especificación clara que guíe la implementación de un sistema de permisos granular que proteja los datos financieros de cada cliente y restrinja las operaciones según el rol asignado.

## Descripción

Esta historia define conceptualmente el sistema de autorización de la plataforma de gestión de fondos de inversión de BTG Pactual. Establece la matriz de permisos para los dos roles definidos (Cliente y Administrador), las reglas de aislamiento de datos (un cliente solo accede a sus propios datos), y el flujo de evaluación de permisos a partir del JWT obtenido en el login. El token JWT (generado según HU 1.2) contiene el userId y rol del usuario, y en cada petición a un endpoint protegido se extrae esta información para evaluar si la operación está permitida.

---

## Criterios de Aceptación

### Escenario 1: Cliente accede a sus propios datos

- **Dado** que un usuario con rol "Cliente" se autenticó y obtuvo un JWT válido
- **Cuando** realiza una operación sobre sus propios recursos (suscripción, cancelación, historial de transacciones) enviando el token en la cabecera `Authorization: Bearer {token}`
- **Entonces** el sistema extrae userId y rol del JWT, verifica que el recurso solicitado pertenece al userId del token, y permite la operación

### Escenario 2: Cliente intenta acceder a datos de otro cliente

- **Dado** que un usuario con rol "Cliente" tiene un JWT válido
- **Cuando** intenta acceder a datos (suscripciones, historial, saldo) de otro cliente diferente al userId de su token
- **Entonces** el sistema rechaza la petición con código HTTP 403 (Forbidden) sin revelar si el recurso del otro cliente existe o no, mostrando un mensaje genérico "Acceso denegado"

### Escenario 3: Administrador accede a datos de cualquier cliente

- **Dado** que un usuario con rol "Administrador" tiene un JWT válido
- **Cuando** solicita ver información de cualquier cliente, gestionar fondos o desbloquear una cuenta
- **Entonces** el sistema permite la operación ya que el rol Administrador tiene permisos amplios sobre todos los recursos de la plataforma

### Escenario 4: Cliente intenta ejecutar operación de administrador

- **Dado** que un usuario con rol "Cliente" tiene un JWT válido
- **Cuando** intenta ejecutar una operación reservada para administradores (gestionar fondos, ver datos de otros clientes, desbloquear usuarios)
- **Entonces** el sistema rechaza la petición con código HTTP 403 (Forbidden) y mensaje "No tiene permisos para realizar esta operación"

### Escenario 5: Petición a endpoint público sin autenticación

- **Dado** que un usuario (registrado o no) realiza una petición
- **Cuando** accede al endpoint de registro o al endpoint de login
- **Entonces** el sistema permite la petición sin validar token ni permisos, ya que estos son los dos únicos endpoints públicos de la plataforma

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Todos los usuarios autenticados de la plataforma (Clientes y Administradores)
- **Permisos requeridos:** Depende del rol — cada rol tiene una matriz de permisos diferenciada
- **Valor de negocio:** Proteger los datos financieros de cada cliente garantizando que solo pueda operar sobre sus propios fondos y transacciones, mientras que los administradores tienen visibilidad completa para gestión y soporte

### Reglas de Negocio

- Dos roles definidos: **Cliente** y **Administrador**
- Los usuarios auto-registrados reciben automáticamente el rol "Cliente"
- El rol "Administrador" se asigna por configuración del sistema o por otro administrador
- **Aislamiento de datos:** Un cliente SOLO puede acceder a sus propios datos (el userId del JWT se compara con el recurso solicitado)
- **Endpoints públicos (sin autenticación):** Solo registro y login
- **Todos los demás endpoints** requieren token JWT válido y evaluación de permisos
- Al denegar acceso a recursos de otro cliente, NO se revela si el recurso existe (prevención de enumeración)
- Todo el proceso debe incluir manejo adecuado de excepciones con mensajes claros

### Matriz de Permisos

| Operación | Cliente | Administrador |
|---|---|---|
| Registrarse (público) | ✅ | ✅ |
| Login (público) | ✅ | ✅ |
| Suscribir fondo propio | ✅ | ✅ |
| Cancelar suscripción propia | ✅ | ✅ |
| Ver historial de transacciones propio | ✅ | ✅ |
| Ver saldo propio | ✅ | ✅ |
| Consultar fondo por ID | ✅ | ✅ |
| Ver datos de otros clientes | ❌ | ✅ |
| Gestionar fondos (crear, editar, eliminar) | ❌ | ✅ |
| Desbloquear cuentas bloqueadas | ❌ | ✅ |
| Ver historial de transacciones de otros | ❌ | ✅ |

### Flujo de Autorización por Petición

```
1. Petición llega al sistema con cabecera Authorization: Bearer {token}
2. Se valida el JWT (firma, expiración) — si falla → 401 (ver HU 1.2)
3. Se extrae userId y rol del JWT
4. Se evalúa contra la matriz de permisos:
   a. ¿El rol tiene permiso para esta operación?  → Si NO → 403
   b. ¿El recurso pertenece al userId? (solo para Cliente) → Si NO → 403
5. Si pasa ambas validaciones → se permite la operación
```

### Catálogo de Excepciones de Autorización

| Caso | Código HTTP | Mensaje |
|---|---|---|
| Rol sin permiso para la operación | 403 Forbidden | "No tiene permisos para realizar esta operación" |
| Cliente accediendo a datos de otro | 403 Forbidden | "Acceso denegado" |
| Token válido pero sin rol requerido | 403 Forbidden | "No tiene permisos para realizar esta operación" |

### Sistemas Externos

No aplica — la autorización es interna basada en el JWT y la matriz de permisos definida.

## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** Autorización en dos capas — Rol en API (`@PreAuthorize`) + Aislamiento de datos en Servicio/Manejador — sobre Arquitectura Hexagonal existente

**Justificación:** La autorización tiene dos dimensiones ortogonales que se resuelven en capas distintas: (1) **Autorización por rol** se resuelve en la capa API con `@PreAuthorize` de Spring Security, como concern del adaptador HTTP ("¿Este rol puede invocar este endpoint?"), activado por `@EnableMethodSecurity` en `SecurityConfig`. (2) **Aislamiento de datos** se resuelve en la capa de servicio/manejador como regla de negocio ("Un cliente solo opera sobre sus propios recursos"). El `clienteId` se extrae del JWT en el controlador y fluye como parámetro en los Comandos/Consultas. Los servicios de dominio validan la propiedad del recurso. Esto mantiene coherencia con el patrón hexagonal donde la lógica de negocio no depende de la capa de seguridad. Decisión crítica: el `clienteId` en operaciones de CLIENTE DEBE provenir del token JWT (SecurityContext), NUNCA del body de la petición, previniendo manipulación de identidad.

**Componentes Afectados:**

- **SecurityConfig.java (Modificación — Menor):** Agregar `@EnableMethodSecurity` para habilitar `@PreAuthorize` en controladores. La clase ya existe por HU 1.2 con SecurityFilterChain, endpoints públicos/protegidos y sesión STATELESS. HU 1.3 solo agrega la anotación de habilitación.
  - Nivel de cambio: Menor

- **ControladorFondo.java (Modificación — Mayor):** Cambio fundamental en el origen del `clienteId`. El método `suscribir()` actualmente toma `clienteId` del body (`SolicitudSuscripcion`). Con HU 1.3, el controlador recibe `Authentication` como parámetro de método (inyectado por Spring Security), extrae `userId` vía `auth.getName()`, y lo usa para construir `FondoComando`. Para rol CLIENTE: `clienteId = auth.getName()` (token JWT, ignora body). Para rol ADMINISTRADOR: `clienteId` puede venir del body (opera en nombre de cualquier cliente). El mismo patrón aplica a futuros endpoints de cancelar y listar vigentes.
  - Nivel de cambio: Mayor (cambia fuente del clienteId, parámetro sensible de seguridad)

- **ControladorAdmin.java (Nuevo):** Controlador REST en `api/controller/` dedicado a operaciones exclusivas de administrador. Endpoint `PUT /api/admin/usuarios/{userId}/desbloquear` — anotado con `@PreAuthorize("hasRole('ADMINISTRADOR')")`, recibe `userId` del path, crea `DesbloqueoComando`, envía al Mediador, retorna `DesbloqueoResultado` con HTTP 200. @RestController, @RequestMapping("/api/admin"), @Tag OpenAPI.

- **DesbloqueoComando.java (Nuevo):** Record en `application/admin/command/` — DTO de entrada para desbloqueo de cuenta. Campo: `userId` (@NotBlank).

- **DesbloqueoResultado.java (Nuevo):** Record en `application/admin/command/` — DTO de salida. Campos: `userId`, `nombre`, `mensaje`.

- **DesbloqueoManejador.java (Nuevo):** `@Component` en `application/admin/command/` que implementa `Manejador<DesbloqueoComando, DesbloqueoResultado>`. Orquesta: buscar cliente por ID → validar que la cuenta está bloqueada → desbloquear cuenta (resetear intentos + desbloquear) → guardar → retornar resultado. Recibe `PuertoRepositorioCliente` por inyección de constructor.

- **Cliente.java (Modificación — Menor):** Agregar método `desbloquearCuenta()` que establece `cuentaBloqueada = false` y `intentosFallidosLogin = 0`. HU 1.2 ya define `bloquearCuenta()`, `incrementarIntentosFallidos()`, `resetearIntentosFallidos()`, `estaBloqueada()`. HU 1.3 agrega el método inverso para desbloqueo administrativo.
  - Nivel de cambio: Menor (un método nuevo sobre estructura ya definida por HU 1.1/1.2)

- **ManejadorExcepcionesGlobal.java (Modificación — Menor):** HU 1.2 ya agrega handlers para 401 y 403 de autenticación. HU 1.3 puede reutilizar el handler 403 existente para denegación de acceso por autorización, o agregar `ExcepcionAccesoDenegado` → HTTP 403 con mensaje "Acceso denegado" para distinguir aislamiento de datos de falta de rol.
  - Nivel de cambio: Menor

**Hitos de Implementación:**

1. **SecurityConfig — @EnableMethodSecurity** — Agregar anotación para habilitar `@PreAuthorize` en controladores
   - Dependencias: SecurityConfig existente (HU 1.2)
2. **Cliente.java — método desbloquearCuenta()** — Agregar comportamiento de desbloqueo administrativo
   - Dependencias: HU 1.2 completada (métodos de bloqueo)
3. **DesbloqueoComando + DesbloqueoResultado + DesbloqueoManejador** — CQRS command completo de desbloqueo en `application/admin/command/`
   - Dependencias: Cliente con desbloquearCuenta(), PuertoRepositorioCliente
4. **ControladorAdmin — endpoint desbloqueo** — `PUT /api/admin/usuarios/{userId}/desbloquear` con @PreAuthorize("hasRole('ADMINISTRADOR')")
   - Dependencias: CQRS desbloqueo, @EnableMethodSecurity
5. **ControladorFondo — aislamiento de datos** — Refactorizar `suscribir()` para extraer clienteId del JWT (Authentication) en vez del body. Patrón para todos los endpoints de operación del cliente
   - Dependencias: SecurityContext funcional (HU 1.2), @EnableMethodSecurity
6. **ManejadorExcepcionesGlobal — ExcepcionAccesoDenegado** — Handler para 403 por violación de aislamiento de datos
   - Dependencias: Definir excepción de acceso denegado

### Validación de Impacto

Se verificó el código fuente real de todos los componentes involucrados:

- **ControladorFondo.java:** Actualmente el método `suscribir()` toma `clienteId` de `SolicitudSuscripcion` (body de la petición). Esto es una vulnerabilidad de seguridad — un cliente podría enviar el ID de otro cliente. HU 1.3 corrige esto extrayendo `clienteId` del JWT. El endpoint `GET /api/fondos/{fondoId}` no necesita aislamiento (consulta de fondo público).
- **FondoComando.java:** Record con `(String clienteId, String fondoId, BigDecimal monto)`. El `clienteId` se mantiene pero su origen en el controlador cambia de body a JWT.
- **ServicioSuscripcion.java:** Usa `clienteId` para buscar cliente y validar saldo. También valida duplicados con `existePorClienteIdYFondoId()`. No requiere cambios — recibe `clienteId` correctamente del comando.
- **FondoManejador.java (command):** Delega directamente a `IServicioSuscripcion.suscribir()` con los campos del comando. No requiere cambios.
- **Suscripcion.java:** Tiene campo `clienteId` que ya permite filtrar por propietario. Base correcta para aislamiento de datos.
- **PuertoRepositorioSuscripcion:** Ya tiene `existePorClienteIdYFondoId()`. Necesitará `buscarPorClienteIdYEstado()` para listar vigentes (HU 2.3), que también sirve para aislamiento.
- **SecurityConfig.java (HU 1.2):** Existe con SecurityFilterChain, sesión STATELESS, endpoints públicos. Solo requiere agregar `@EnableMethodSecurity`.
- **JwtFiltroAutenticacion.java (HU 1.2):** Establece `SecurityContext` con `UsernamePasswordAuthenticationToken`. Debe asegurar que las authorities incluyan prefijo `ROLE_` para compatibilidad con `hasRole()`.
- **Tests existentes (ControladorFondoE2ETest):** Ya necesitan adaptación por HU 1.2 (Spring Security). HU 1.3 agrega la necesidad de que los tests provean un usuario autenticado con rol correcto.

### Notas Técnicas (Si aplica)

- **Prefijo ROLE_ en authorities:** Spring Security `hasRole('ADMINISTRADOR')` busca `ROLE_ADMINISTRADOR` en las authorities. El `JwtFiltroAutenticacion` (HU 1.2) debe construir `SimpleGrantedAuthority("ROLE_" + rol)` al crear el `UsernamePasswordAuthenticationToken`. Esto es un requisito de integración entre HU 1.2 y HU 1.3.
- **Origen del clienteId — patrón de resolución:** En el controlador, el patrón recomendado es: `String clienteId = auth.getAuthorities().stream().anyMatch(a -> a.getAuthority().equals("ROLE_ADMINISTRADOR")) ? solicitud.clienteId() : auth.getName()`. Esto permite que ADMIN opere en nombre de cualquier cliente mientras CLIENTE solo opera sobre sí mismo.
- **Deuda técnica de DTOs en api/dto/:** `SolicitudSuscripcion` en `api/dto/` incluye `clienteId` como campo. Con aislamiento de datos, este campo se vuelve irrelevante para CLIENTE (se ignora a favor del JWT). Podría eliminarse o mantenerse opcional. La migración de DTOs a capa de aplicación no es alcance de HU 1.3.
- **AccessDeniedException de Spring Security:** `@PreAuthorize` lanza `AccessDeniedException` cuando falla. Spring Security la mapea automáticamente a 403. Si se necesita personalizar el mensaje, se puede configurar un `AccessDeniedHandler` en `SecurityConfig` o capturar `AccessDeniedException` en `ManejadorExcepcionesGlobal`.

## Refinamiento Técnico (Developer)

### Consideraciones Generales

**Basado en análisis arquitectónico:**
Sección "Análisis Arquitectónico (Arquitecto)" de esta misma historia — 6 hitos de implementación, 6 componentes afectados (1 nuevo: ControladorAdmin; 3 modificados: SecurityConfig, ControladorFondo, ManejadorExcepcionesGlobal; 2 ya implementados en HU 1.2: Cliente.desbloquearCuenta() y CQRS desbloqueo completo). Autorización en dos capas: Rol en API (`@PreAuthorize`) + Aislamiento de datos en Controlador (clienteId del JWT).

**Nivel de complejidad:**
MEDIA - Modifica componente transversal de seguridad (SecurityConfig, controladores, exception handler) + refactoring del origen del clienteId en ControladorFondo + nuevo controlador admin con `@PreAuthorize`. La mayoría de componentes base ya existen por HU 1.2 (SecurityConfig con STATELESS, JwtFiltroAutenticacion con prefijo `ROLE_`, DesbloqueoManejador, Cliente.desbloquearCuenta()). No hay dependencias nuevas ni migraciones.

**Riesgos técnicos conocidos:**
- Breaking change en endpoint de desbloqueo: mover de `PUT /api/auth/desbloqueo/{clienteId}` a `PUT /api/admin/usuarios/{userId}/desbloquear`. Los tests E2E existentes de desbloqueo en `ControladorAutenticacionE2ETest` dejan de funcionar. Mitigación: eliminar tests obsoletos y crear nuevos en `ControladorAdminE2ETest` en la misma fase.
- Cambio en origen de clienteId en ControladorFondo: tests E2E existentes envían clienteId en body de `SolicitudSuscripcion`. Tras refactoring, para CLIENTE el body se ignora y se usa JWT. Mitigación: adaptar tests existentes para usar token JWT real con `auth.getName()` como clienteId.
- `AccessDeniedException` vs `ExcepcionCuentaBloqueada`: ambas producen HTTP 403. Spring Security lanza `AccessDeniedException` para `@PreAuthorize`, y `ManejadorExcepcionesGlobal` ya maneja `ExcepcionCuentaBloqueada` como 403. Mitigación: mensajes diferenciados — "No tiene permisos para realizar esta operación" vs "Cuenta bloqueada".
- Reconciliación con análisis arquitectónico: el arquitecto marcó DesbloqueoComando, DesbloqueoResultado, DesbloqueoManejador y `Cliente.desbloquearCuenta()` como "Nuevo", pero ya existen (implementados en HU 1.2 en `application/desbloqueo/command/`). Las tareas son de MOVER el endpoint a nuevo controlador con autorización, no de crear CQRS desde cero.

**Patrones y convenciones del equipo:**
- Records para Comandos/Consultas/Resultados (DTOs en capa `application/`, NO en `api/dto/`)
- Inyección por constructor, nunca `@Autowired` en campo
- `@PreAuthorize` para autorización por rol en controladores (nuevo en HU 1.3)
- `Authentication` inyectado como parámetro de método en controlador para extraer userId del JWT
- Modelos de dominio con lógica encapsulada (no anémicos)
- `ExcepcionDominio` para reglas de negocio (→422), excepciones específicas para HTTP diferenciados
- Mediador auto-registra handlers por tipo — solo crear `@Component`
- Prefijo `ROLE_` en `SimpleGrantedAuthority` de `JwtFiltroAutenticacion` — compatible con `hasRole()`

**Dependencias nuevas a instalar:**
Ninguna — `spring-boot-starter-security` (HU 1.2) ya incluye `@EnableMethodSecurity` y `@PreAuthorize`. `spring-security-test` ya está en `testImplementation`.

**Estrategia de testing:**
JUnit 5 + MockMvc + Mockito + spring-security-test | Tests E2E (ControladorAdmin: 3 escenarios, ControladorFondo aislamiento: 3 escenarios, adaptación tests existentes: desbloqueo + suscripción) | Cobertura: AC1-AC5 con tests directos | Builders: ClienteBuilder (reutilizar), RegistroComandoBuilder (reutilizar), LoginComandoBuilder (reutilizar)
