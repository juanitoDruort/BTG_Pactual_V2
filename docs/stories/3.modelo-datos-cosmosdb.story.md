# Historia #3: Diseño del modelo de datos NoSQL en Azure Cosmos DB

**Estado:** Borrador (PO)  
**Tipo:** Historia Técnica (Diseño)

---

## Historia de Usuario

**Como** equipo de desarrollo de la plataforma BTG Pactual,  
**Quiero** contar con un diseño documentado del modelo de datos NoSQL en Azure Cosmos DB que soporte todas las operaciones del sistema,  
**Para** tener una especificación clara de colecciones, estructura de documentos, partition keys, índices, estrategia de denormalización y datos semilla que guíe la implementación de la capa de persistencia.

## Descripción

El sistema actualmente opera con persistencia en memoria (ConcurrentHashMap) a través de adaptadores in-memory en la capa de infraestructura. Esta historia técnica define el diseño completo del modelo de datos para Azure Cosmos DB (API NoSQL), incluyendo la estructura de cada colección/contenedor, la estrategia de denormalización, las partition keys optimizadas para los patrones de acceso identificados, la política de indexación, y los datos semilla necesarios para inicialización. El modelo debe contemplar las 3 entidades de dominio actuales (Cliente, Fondo, Suscripción) con la extensión del modelo de Cliente definida en HU 1.1 (campos de seguridad). El entorno de desarrollo utiliza el emulador de Cosmos DB en Docker. Este diseño es el artefacto de referencia que el equipo de desarrollo usará para implementar los nuevos adaptadores de persistencia.

## Criterios de Aceptación

### Escenario 1: Definición de contenedores y estructura de documentos

- **Dado** que se requiere un modelo de datos para Cosmos DB
- **Cuando** se revisa el documento de diseño
- **Entonces** se encuentra definido cada contenedor (colección) con su nombre, estructura JSON del documento (campos, tipos de datos, campos embebidos), y la justificación de la estrategia de denormalización aplicada

### Escenario 2: Definición de partition keys por contenedor

- **Dado** que Cosmos DB requiere partition keys para distribución de datos
- **Cuando** se revisa el documento de diseño
- **Entonces** cada contenedor tiene definida su partition key con la justificación basada en los patrones de acceso identificados (cliente por ID/email, fondo por ID, suscripciones activas por cliente, verificación de duplicado)

### Escenario 3: Política de indexación definida

- **Dado** que los patrones de consulta están identificados
- **Cuando** se revisa el documento de diseño
- **Entonces** cada contenedor tiene definida su política de indexación (índices incluidos, excluidos, índices compuestos si aplica) optimizada para los 5 patrones de acceso prioritarios

### Escenario 4: Modelo del Cliente incluye campos extendidos de HU 1.1

- **Dado** que HU 1.1 extiende el modelo de Cliente con campos de seguridad
- **Cuando** se revisa la estructura del documento Cliente
- **Entonces** incluye todos los campos: id, nombre, saldo, email, telefono, documentoIdentidad, contraseña (hash), rol, bloqueado, intentosFallidos, preferenciaNotificacion

### Escenario 5: Estrategia de denormalización documentada

- **Dado** que se decidió utilizar denormalización como estrategia de modelado
- **Cuando** se revisa el documento de diseño
- **Entonces** se documenta qué datos se embeben en cada contenedor, la justificación de cada decisión de embebido, y las implicaciones en actualización de datos denormalizados

### Escenario 6: Datos semilla completos y consistentes

- **Dado** que el sistema necesita datos iniciales para funcionar
- **Cuando** se revisan los datos semilla en el documento
- **Entonces** se incluyen los 2 clientes existentes (con todos los campos extendidos completados), los 5 fondos del catálogo, y cualquier dato adicional necesario para que las semillas sean consistentes con el modelo diseñado

### Escenario 7: Cobertura de patrones de acceso

- **Dado** que existen 5 patrones de consulta prioritarios
- **Cuando** se revisa el documento de diseño
- **Entonces** cada patrón tiene documentada la query correspondiente en Cosmos DB y se verifica que el diseño (partition key + índices) lo soporta eficientemente:
  - Buscar cliente por ID
  - Buscar cliente por email
  - Buscar fondo por ID
  - Listar suscripciones activas de un cliente
  - Verificar duplicado (clienteId + fondoId + estado ACTIVO)

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Equipo de desarrollo (desarrolladores, arquitecto) que usará este artefacto como referencia para implementar los adaptadores de Cosmos DB
- **Permisos requeridos:** No aplica — es un artefacto de documentación técnica
- **Valor de negocio:** Proveer la especificación técnica que permite migrar de persistencia in-memory a una base de datos real (Cosmos DB), habilitando la evolución del sistema hacia un entorno productivo con datos persistentes

### Reglas de Negocio y Decisiones Técnicas

- **Motor:** Azure Cosmos DB con API NoSQL
- **Entorno de desarrollo:** Emulador de Cosmos DB en Docker
- **Estrategia de modelado:** Denormalización — embeber datos relacionados en documentos para optimizar lecturas
- **Entregable:** Solo diseño documental (no implementación de código)
- **Adaptadores in-memory:** No se modifican — siguen funcionando para tests y desarrollo sin Docker
- **Patrones de consulta:** Los 5 identificados son el alcance actual; se pueden ajustar a futuro
- **Campos extendidos:** El diseño incluye la extensión del modelo de Cliente de HU 1.1

## Contexto y Referencias

**Arquitectura:** Arquitectura Hexagonal + CQRS + Mediador según [ARQUITECTURA.md](../../ARQUITECTURA.md). El modelo de datos Cosmos DB se alinea con los puertos de repositorio definidos en la capa de dominio (`PuertoRepositorioCliente`, `PuertoRepositorioFondo`, `PuertoRepositorioSuscripcion`). Los adaptadores in-memory actuales no se modifican — este diseño es la base para futuros adaptadores de Cosmos DB en `infrastructure/adapter/out/persistence/`.

**Historias relacionadas:**
- HU 1.1 (Registro — define extensión del modelo de Cliente que se incluye en este diseño)
- HU 2.1-2.4 (Operaciones de suscripción — definen los patrones de acceso a soportar)
- HU 1.4 (Encriptación — la contraseña se almacena como hash, no en texto plano)

**Lecciones aprendidas:**
- Los puertos de repositorio actuales definen el contrato mínimo; el diseño NoSQL debe soportar al menos esas operaciones
- `PuertoRepositorioSuscripcion` carece de `buscarPorClienteId()` — el diseño debe contemplar este patrón (requerido por HU 2.3)
- La verificación de duplicados actual filtra por estado ACTIVO — esto debe reflejarse en la estrategia de indexación
