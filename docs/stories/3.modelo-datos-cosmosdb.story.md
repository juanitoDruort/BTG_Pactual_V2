# Historia #3: Diseño del modelo de datos NoSQL en Azure Cosmos DB

**Estado:** Analizado (Arquitecto)  
**Tipo:** Historia Técnica (Diseño)

---

## Historia de Usuario

**Como** equipo de desarrollo de la plataforma BTG Pactual,  
**Quiero** contar con un diseño documentado del modelo de datos NoSQL en Azure Cosmos DB que soporte todas las operaciones del sistema,  
**Para** tener una especificación clara de colecciones, estructura de documentos, partition keys, índices, estrategia de denormalización y datos semilla que guíe la implementación de la capa de persistencia.

## Descripción

El sistema actualmente opera con persistencia en memoria (ConcurrentHashMap) a través de adaptadores in-memory en la capa de infraestructura. Esta historia técnica define el diseño completo del modelo de datos para Azure Cosmos DB (API NoSQL), incluyendo la estructura de cada colección/contenedor, la estrategia de denormalización, las partition keys optimizadas para los patrones de acceso identificados, la política de indexación, y los datos semilla necesarios para inicialización. El modelo debe contemplar las 3 entidades de dominio actuales (Cliente, Fondo, Suscripción) con la extensión del modelo de Cliente definida en HU 1.1 (campos de seguridad). El entorno de desarrollo utiliza el emulador de Cosmos DB en Docker. Este diseño es el artefacto de referencia que el equipo de desarrollo usará para implementar los nuevos adaptadores de persistencia.

### Contenedor `clientes`

**Partition Key:** `/id`

**Justificación:** Point read por ID es la operación más frecuente (cada suscripción/cancelación consulta al cliente). Query por email (login/registro) es cross-partition pero optimizada por índice en email. Cardinalidad baja-media de clientes.

**Estructura del documento:**

```json
{
  "id": "cliente-1",
  "nombre": "Juan Rodriguez",
  "saldo": 500000,
  "email": "juan.rodriguez@email.com",
  "telefono": "+573001234567",
  "documentoIdentidad": "1234567890",
  "contrasena": "$2a$10$...(hash BCrypt)",
  "rol": "CLIENTE",
  "bloqueado": false,
  "intentosFallidos": 0,
  "preferenciaNotificacion": "EMAIL"
}
```

**Política de indexación:**

```json
{
  "includedPaths": [
    { "path": "/email/?" },
    { "path": "/rol/?" }
  ],
  "excludedPaths": [
    { "path": "/contrasena/?" },
    { "path": "/documentoIdentidad/?" },
    { "path": "/nombre/?" },
    { "path": "/*" }
  ]
}
```

**Unique key policy:** `{ "uniqueKeys": [{ "paths": ["/email"] }] }` — Unicidad de email a nivel de base de datos.

**Queries:**

| Operación | Query Cosmos DB | Eficiencia |
|---|---|---|
| `buscarPorId("cliente-1")` | `ReadItem("cliente-1", PK("cliente-1"))` | 1 RU (point read) |
| `guardar(cliente)` | `UpsertItem(doc, PK(doc.id))` | ~5 RU (upsert) |
| `existePorEmail(email)` | `SELECT VALUE COUNT(1) FROM c WHERE c.email = @email` | Cross-partition, índice en email |
| `buscarPorEmail(email)` | `SELECT * FROM c WHERE c.email = @email` | Cross-partition, índice en email |

### Contenedor `fondos`

**Partition Key:** `/id`

**Justificación:** Único patrón es point read por ID. Catálogo inmutable de 5 documentos.

**Estructura del documento:**

```json
{
  "id": "fondo-1",
  "nombre": "FPV_BTG_PACTUAL_RECAUDADORA",
  "montoMinimo": 75000,
  "categoria": "FPV"

```

**Política de indexación:** Default (todo incluido) — contenedor diminuto (5 docs), no justifica optimización.

**Queries:**

| Operación | Query Cosmos DB | Eficiencia |
|---|---|---|
| `buscarPorId("fondo-1")` | `ReadItem("fondo-1", PK("fondo-1"))` | 1 RU (point read) |

### Contenedor `suscripciones`

**Partition Key:** `/clienteId`

**Justificación:** Patrón dominante es listar suscripciones activas por cliente (single-partition). Verificación de duplicados también single-partition. Point read por ID de suscripción requiere clienteId adicional (disponible desde JWT).

**Estructura del documento (con denormalización):**

```json
{
  "id": "uuid-suscripcion-1",
  "clienteId": "cliente-1",
  "fondoId": "fondo-1",
  "nombreFondo": "FPV_BTG_PACTUAL_RECAUDADORA",
  "monto": 75000,
  "estado": "ACTIVO",
  "fechaSuscripcion": "2026-03-12T10:30:00"
}
```

**Denormalización `nombreFondo`:**
- HU 2.3 necesita nombre del fondo al listar vigentes. Sin embebido = N queries a `fondos`.
- Fondos son catálogo inmutable → sin riesgo de inconsistencia.
- `fondoId` se mantiene como referencia canónica.

**Política de indexación:**

```json
{
  "includedPaths": [
    { "path": "/estado/?" },
    { "path": "/fondoId/?" }
  ],
  "excludedPaths": [
    { "path": "/nombreFondo/?" },
    { "path": "/monto/?" },
    { "path": "/fechaSuscripcion/?" },
    { "path": "/*" }
  ],
  "compositeIndexes": [
    [
      { "path": "/fondoId", "order": "ascending" },
      { "path": "/estado", "order": "ascending" }
    ]
  ]
}
```

**Queries:**

| Operación | Query Cosmos DB | Eficiencia |
|---|---|---|
| `guardar(suscripcion)` | `UpsertItem(doc, PK(doc.clienteId))` | ~5 RU (single-partition) |
| `existePorClienteIdYFondoId(c, f)` | `SELECT VALUE COUNT(1) FROM c WHERE c.clienteId=@cid AND c.fondoId=@fid AND c.estado='ACTIVO'` | Single-partition, índice compuesto |
| `buscarPorId(id)` | `SELECT * FROM c WHERE c.id=@id AND c.clienteId=@cid` | Single-partition (clienteId del JWT) |
| `buscarPorClienteIdYEstado(c, ACTIVO)` | `SELECT * FROM c WHERE c.clienteId=@cid AND c.estado='ACTIVO'` | Single-partition, índice en estado |

### Hitos de Implementación

| Hito | Entregable | Verificación |
|---|---|---|
| **H1** — Contenedor `clientes` | Estructura, PK `/id`, indexación selectiva, unique key `/email` | Cubre operaciones #1-#4 |
| **H2** — Contenedor `fondos` | Estructura, PK `/id`, indexación default | Cubre operación #5 |
| **H3** — Contenedor `suscripciones` | Estructura con denormalización `nombreFondo`, PK `/clienteId`, índice compuesto | Cubre operaciones #6-#9 |
| **H4** — Datos semilla | 2 clientes (campos extendidos), 5 fondos | Consistentes con modelo |
| **H5** — Validación de patrones | 9 operaciones con query + estimación RU | Cada patrón documentado |

### Decisiones Arquitectónicas

1. **PK `/clienteId` en suscripciones** — Optimiza patrón más frecuente (listar activas) como single-partition. Point read por ID requiere clienteId adicional (disponible desde JWT).
2. **Denormalización `nombreFondo`** — Evita N lookups al listar vigentes. Catálogo de fondos inmutable = sin riesgo de inconsistencia.
3. **Indexación selectiva en `clientes`** — Solo `email` y `rol`. Excluye `contrasena`, `documentoIdentidad` (nunca como filtro). Reduce costos RU en escritura.
4. **Unique key `/email`** — Defensa en profundidad junto con `existePorEmail()`.
5. **3 contenedores separados** — Entidades con ciclos de vida distintos. Embeber suscripciones en Cliente = unbounded arrays (anti-pattern Cosmos DB).
6. **`saldo` como número en documento** — HU 1.4 (AES-256-GCM) se aplica en el adaptador de persistencia (encripta al guardar, desencripta al leer).
