# Historia #1.2: Definición del proceso de autenticación con JWT y política de bloqueo de cuentas

**Estado:** Lista para Revisión

---

## Historia de Usuario

**Como** administrador de la plataforma BTG Pactual,  
**Quiero** contar con una definición conceptual completa del proceso de autenticación mediante JWT y la política de bloqueo de cuentas,  
**Para** que el equipo de desarrollo tenga una especificación clara que guíe la implementación de un mecanismo de autenticación seguro con gestión de sesiones stateless y protección contra accesos no autorizados.

## Descripción

Esta historia define conceptualmente el flujo de autenticación de la plataforma de gestión de fondos de inversión de BTG Pactual. El sistema debe implementar autenticación basada en JWT (JSON Web Token) stateless con tokens de expiración configurable (default 5 minutos). El endpoint de login recibe credenciales (email y contraseña), valida al usuario y retorna un JWT que el cliente utiliza para consumir todos los demás endpoints protegidos enviándolo en la cabecera `Authorization: Bearer {token}`. Incluye política de bloqueo tras intentos fallidos y proceso de desbloqueo por administrador.

## Criterios de Aceptación

### Escenario 1: Login exitoso — credenciales válidas retornan JWT

- **Dado** que un usuario registrado con cuenta activa (no bloqueada) accede al endpoint público de login
- **Cuando** envía su email y contraseña correctos
- **Entonces** el sistema valida las credenciales, resetea el contador de intentos fallidos a 0, genera un JWT stateless con claims (userId, rol, fecha de expiración) y lo retorna en el cuerpo de la respuesta con código HTTP 200. Este token es el que el usuario debe enviar en la cabecera `Authorization: Bearer {token}` para consumir todos los endpoints protegidos

### Escenario 2: Login fallido — credenciales inválidas

- **Dado** que un usuario envía credenciales al endpoint de login
- **Cuando** el email no existe o la contraseña es incorrecta
- **Entonces** el sistema incrementa el contador de intentos fallidos del usuario, retorna un error genérico "Credenciales inválidas" (sin revelar si el problema es el email o la contraseña) con código HTTP 401 (Unauthorized)

### Escenario 3: Bloqueo automático tras intentos fallidos

- **Dado** que un usuario ha acumulado 2 intentos fallidos consecutivos de login
- **Cuando** falla un tercer intento de autenticación (configurable, default 3)
- **Entonces** la cuenta se bloquea automáticamente, se retorna un mensaje "Cuenta bloqueada. Contacte al administrador para desbloquearla" con código HTTP 403 (Forbidden), y todos los intentos posteriores de login con esas credenciales retornan el mismo mensaje de bloqueo sin incrementar el contador

### Escenario 4: Acceso a endpoint protegido con token válido

- **Dado** que un usuario se autenticó exitosamente y obtuvo un JWT
- **Cuando** realiza una petición a cualquier endpoint protegido enviando el token en la cabecera `Authorization: Bearer {token}` y el token no ha expirado
- **Entonces** el sistema valida el token, extrae userId y rol, y permite que la petición continúe al proceso de autorización (HU 1.3)

### Escenario 5: Acceso a endpoint protegido con token expirado

- **Dado** que un usuario tiene un JWT cuyo tiempo de expiración (5 minutos por defecto) ya venció
- **Cuando** intenta acceder a un endpoint protegido con dicho token
- **Entonces** el sistema rechaza la petición con mensaje "Token expirado. Inicie sesión nuevamente" con código HTTP 401 (Unauthorized)

### Escenario 6: Acceso a endpoint protegido sin token o token malformado

- **Dado** que se recibe una petición a un endpoint protegido
- **Cuando** la cabecera `Authorization` está ausente, vacía, o el token no es un JWT válido
- **Entonces** el sistema rechaza la petición con mensaje "Token de autenticación requerido" o "Token inválido" con código HTTP 401 (Unauthorized)

### Escenario 7: Desbloqueo de cuenta por administrador

- **Dado** que un administrador autenticado identifica una cuenta bloqueada
- **Cuando** ejecuta la acción de desbloqueo para dicho usuario
- **Entonces** el sistema desbloquea la cuenta, resetea el contador de intentos fallidos a 0, y el usuario puede volver a intentar login normalmente

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Cualquier usuario registrado (Cliente o Administrador) que necesita autenticarse para acceder a la plataforma
- **Permisos requeridos:** Ninguno para login (endpoint público). Rol Administrador para desbloquear cuentas
- **Valor de negocio:** Garantizar que solo usuarios legítimos accedan a la plataforma de gestión de fondos, protegiendo datos financieros sensibles de los clientes. El token JWT permite un modelo stateless escalable

### Reglas de Negocio

- Autenticación mediante JWT stateless (no sesiones en servidor)
- El endpoint de login es público — uno de los dos únicos endpoints sin autenticación (junto con registro)
- Credenciales de login: email + contraseña
- El login retorna un JWT en el cuerpo de la respuesta
- El token JWT incluye claims: userId, rol (CLIENTE/ADMINISTRADOR), fecha de expiración
- Expiración del token: 5 minutos por defecto (valor configurable)
- El token se envía en cabecera `Authorization: Bearer {token}` en cada petición a endpoints protegidos
- Bloqueo automático tras 3 intentos fallidos consecutivos (valor configurable)
- El contador de intentos fallidos se resetea tras un login exitoso
- Solo un administrador puede desbloquear una cuenta bloqueada
- NO se implementa autenticación de dos factores (2FA) — decisión explícita del stakeholder
- Todo el proceso debe incluir manejo adecuado de excepciones con mensajes claros

### Estructura del Token JWT

| Claim | Descripción |
|---|---|
| sub | userId del usuario autenticado |
| rol | Rol del usuario (CLIENTE o ADMINISTRADOR) |
| iat | Fecha/hora de emisión del token |
| exp | Fecha/hora de expiración (iat + 5 minutos configurable) |

### Catálogo de Excepciones de Autenticación

| Caso | Código HTTP | Mensaje |
|---|---|---|
| Credenciales inválidas | 401 Unauthorized | "Credenciales inválidas" |
| Cuenta bloqueada | 403 Forbidden | "Cuenta bloqueada. Contacte al administrador" |
| Token expirado | 401 Unauthorized | "Token expirado. Inicie sesión nuevamente" |
| Token malformado | 401 Unauthorized | "Token inválido" |
| Token ausente | 401 Unauthorized | "Token de autenticación requerido" |

### Sistemas Externos

No aplica — la autenticación es completamente interna usando JWT firmado localmente.

## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** Arquitectura Hexagonal + CQRS + Mediador (extensión) + Filtro de Seguridad Transversal

**Justificación:** El flujo de login encaja naturalmente como un nuevo Command (`LoginComando → LoginManejador`) que se registra automáticamente en el Mediador — idéntico al patrón ya validado en registro (HU 1.1) y suscripción de fondos. La validación de JWT en endpoints protegidos se implementa como filtro transversal (`OncePerRequestFilter`) en la cadena de seguridad de Spring Security, coherente con la arquitectura documentada en ARQUITECTURA.md donde la seguridad se resuelve antes de llegar al controlador. La generación/validación de tokens se encapsula en infraestructura (`JwtProveedor`) sin contaminar el dominio. Los DTOs de login viven en la capa de aplicación, consistente con la decisión arquitectónica establecida en HU 1.1.

**Componentes Afectados:**

- **LoginComando.java (Nuevo):** Record en `application/login/command/` que sirve como DTO de entrada para el endpoint de login. Campos: `email` (@NotBlank, @Email) y `contrasena` (@NotBlank). El controlador lo recibe directamente como `@RequestBody @Valid`.

- **LoginResultado.java (Nuevo):** Record en `application/login/command/` que sirve como DTO de salida del login. Campo: `token` (String JWT). Retornado directamente por el controlador como cuerpo de respuesta HTTP 200.

- **LoginManejador.java (Nuevo):** `@Component` en `application/login/command/` que implementa `Manejador<LoginComando, LoginResultado>`. Orquesta el flujo completo: buscar usuario por email → verificar cuenta no bloqueada (o 403) → verificar contraseña con PasswordEncoder → si fallo: incrementar intentos, evaluar bloqueo (≥3), retornar 401 → si éxito: resetear intentos a 0, generar JWT vía `JwtProveedor`, retornar `LoginResultado`. Recibe `PuertoRepositorioCliente` y `JwtProveedor` por inyección de constructor.

- **JwtProveedor.java (Nuevo):** Componente en `infrastructure/security/` responsable de generar y validar tokens JWT. Métodos: `generarToken(String userId, String rol)` → String JWT con claims (sub, rol, iat, exp); `validarToken(String token)` → boolean; `extraerUserId(String token)` → String; `extraerRol(String token)` → String. Lee configuración de `application.properties` (clave secreta, expiración). Usa librería `io.jsonwebtoken:jjwt`.

- **JwtFiltroAutenticacion.java (Nuevo):** Filtro en `infrastructure/security/` que extiende `OncePerRequestFilter`. Se ejecuta en cada petición a endpoints protegidos. Extrae token de cabecera `Authorization: Bearer {token}`, valida firma y expiración vía `JwtProveedor`, establece `SecurityContext` con `UsernamePasswordAuthenticationToken` (userId, rol). Si token inválido/expirado/ausente → responde 401 directamente sin continuar la cadena.

- **DetallesUsuarioServicio.java (Nuevo):** Componente en `infrastructure/security/` que implementa `UserDetailsService` de Spring Security. Carga usuario por email desde `PuertoRepositorioCliente.buscarPorEmail()`. Requerido por la infraestructura de Spring Security para integración estándar.

- **SecurityConfig.java (Nuevo):** Clase `@Configuration` en `infrastructure/config/` que define el `SecurityFilterChain`. Configura: endpoints públicos (`/api/auth/registro`, `/api/auth/login`, Swagger/OpenAPI) → `permitAll()`; todos los demás → `authenticated()`. Registra `JwtFiltroAutenticacion` antes de `UsernamePasswordAuthenticationFilter`. Deshabilita CSRF (API stateless). Configura política de sesión `STATELESS`. Expone bean `PasswordEncoder` (BCrypt).

- **ControladorAutenticacion.java (Modificación — Menor):** Agregar endpoint `POST /api/auth/login` — recibe `LoginComando` como `@RequestBody @Valid`, envía al Mediador, retorna `LoginResultado` con HTTP 200. Se extiende el controlador ya definido en HU 1.1 (que tiene `/api/auth/registro`).
  - Nivel de cambio: Menor

- **Cliente.java (Modificación — Menor):** Agregar métodos de dominio para lógica de bloqueo: `incrementarIntentosFallidos()`, `bloquearCuenta()`, `resetearIntentosFallidos()`, `estaBloqueada()`. Estos métodos encapsulan la lógica de negocio de bloqueo dentro del modelo de dominio (los campos `cuentaBloqueada` e `intentosFallidosLogin` ya están definidos en HU 1.1).
  - Nivel de cambio: Menor (campos ya existen por HU 1.1; solo se agregan métodos de comportamiento)

- **ManejadorExcepcionesGlobal.java (Modificación — Menor):** Agregar handlers para las excepciones de autenticación: `ExcepcionCredencialesInvalidas` → HTTP 401, `ExcepcionCuentaBloqueada` → HTTP 403, `ExcepcionTokenInvalido` → HTTP 401. Crear excepciones específicas que extiendan `ExcepcionDominio` o usar un mecanismo de código/tipo en `ExcepcionDominio` existente.
  - Nivel de cambio: Menor

- **build.gradle (Modificación — Menor):** Agregar dependencias: `spring-boot-starter-security` (reemplaza el `spring-security-crypto` de HU 1.1 ya que lo incluye), `io.jsonwebtoken:jjwt-api:0.12.6`, `io.jsonwebtoken:jjwt-impl:0.12.6` (runtime), `io.jsonwebtoken:jjwt-jackson:0.12.6` (runtime), `spring-security-test` (testImplementation).
  - Nivel de cambio: Menor

  - Nivel de cambio: Menor

### Validación de Impacto

Se verificó el código fuente real de todos los componentes involucrados:

- **Cliente.java:** Actualmente tiene solo 3 campos (id, nombre, saldo) y un constructor simple. HU 1.1 lo extiende con campos de seguridad (cuentaBloqueada, intentosFallidosLogin, contrasenaHash, rol). HU 1.2 solo agrega métodos de comportamiento sobre esos campos — no modifica estructura.
- **PuertoRepositorioCliente:** Solo tiene `buscarPorId()` y `guardar()`. HU 1.1 agrega `buscarPorEmail()` y `existePorEmail()`. HU 1.2 consume `buscarPorEmail()` en el LoginManejador — sin modificaciones adicionales al puerto.
- **ManejadorExcepcionesGlobal:** Actualmente maneja `ExcepcionDominio` → 422 y `MethodArgumentNotValidException` → 400. HU 1.1 agrega 409. HU 1.2 agrega 401 y 403 — extensión incremental sin conflictos.
- **build.gradle:** No tiene dependencias de seguridad. HU 1.1 agrega `spring-security-crypto`. HU 1.2 lo reemplaza por `spring-boot-starter-security` (que incluye crypto) + JJWT.
- **application.properties:** Solo tiene configuración base (app name, Swagger). HU 1.2 agrega propiedades JWT.
- **ControladorFondo:** Actualmente usa DTOs de `api/dto/` (SolicitudSuscripcion, RespuestaFondo, RespuestaSuscripcion) — esto es deuda técnica preexistente fuera del alcance de HU 1.2. Al agregar Spring Security, estos endpoints quedarán protegidos automáticamente.
- **Tests existentes (ControladorFondoE2ETest):** Al activar Spring Security, los tests fallarán sin token. Deben adaptarse con `@WithMockUser` o token de prueba como parte del hito 6 (SecurityConfig).
- **No existen carpetas `infrastructure/security/` ni `infrastructure/config/`** — todas son creaciones nuevas sin conflicto con código existente.

### Notas Técnicas (Si aplica)

- **Reemplazo de spring-security-crypto por starter-security:** En HU 1.1 se definió agregar solo `spring-security-crypto` para BCrypt. Al implementar HU 1.2, `spring-boot-starter-security` incluye crypto, por lo que se reemplaza la dependencia parcial por la completa. Si HU 1.1 se implementa primero, el desarrollador ajustará en HU 1.2.
- **Impacto de autoconfiguración de Spring Security:** Al agregar `spring-boot-starter-security`, Spring Boot protege TODOS los endpoints por defecto. Es crítico que `SecurityConfig.java` se configure en el mismo sprint/tarea que se agrega la dependencia para evitar bloqueo de la aplicación.
- **Clave JWT en producción:** La propiedad `jwt.secret` en `application.properties` es para desarrollo. En producción debe usarse una variable de entorno o vault de secretos. La clave debe tener mínimo 256 bits (32 caracteres) para HMAC-SHA256.
- **Deuda técnica de DTOs en api/dto/:** El ControladorFondo actual usa `SolicitudSuscripcion`, `RespuestaFondo` y `RespuestaSuscripcion` en `api/dto/`. Esto contradice la decisión arquitectónica de DTOs en capa de aplicación, pero su migración no es alcance de HU 1.2. Los nuevos componentes (LoginComando/LoginResultado) siguen la convención correcta.
- **Desbloqueo por administrador (Escenario 7):** El endpoint de desbloqueo para administrador puede implementarse como un command adicional (`DesbloqueoComando → DesbloqueoManejador`) siguiendo el mismo patrón CQRS. Se recomienda incluirlo en el hito 7 como extensión del LoginManejador o como command independiente.

## Refinamiento Técnico (Developer)

### Consideraciones Generales

**Basado en análisis arquitectónico:**
Sección "Análisis Arquitectónico (Arquitecto)" de esta misma historia — 9 hitos de implementación, 12 componentes afectados (7 nuevos: LoginComando, LoginResultado, LoginManejador, JwtProveedor, JwtFiltroAutenticacion, DetallesUsuarioServicio, SecurityConfig; 4 modificados: Cliente, ControladorAutenticacion, ManejadorExcepcionesGlobal, build.gradle/application.properties). Componente transversal de seguridad que impacta todos los endpoints de la aplicación.

**Nivel de complejidad:**
ALTA - Componente transversal de seguridad (Spring Security + JWT) que protege automáticamente TODOS los endpoints existentes al agregar `spring-boot-starter-security`. Requiere 7 clases nuevas en 2 paquetes nuevos (`infrastructure/security/`, `infrastructure/config/`), reemplazo de dependencia (`spring-security-crypto` → `spring-boot-starter-security`), adaptación obligatoria de 17+ tests existentes para funcionar con Spring Security activo, y coordinación entre 5 capas (domain, application, infrastructure/security, infrastructure/config, api).

**Riesgos técnicos conocidos:**
- Al agregar `spring-boot-starter-security`, Spring Boot protege TODOS los endpoints por defecto (retorna 401). Es crítico que `SecurityConfig.java` se implemente en la misma fase en que la dependencia se activa. Mitigación: Fase 0 agrega dependencias; Fase 3 configura SecurityConfig y adapta tests existentes antes de continuar.
- Los 17 tests existentes (`ControladorFondoE2ETest` 12 + `ControladorAutenticacionE2ETest` 5) fallarán al activar Spring Security sin configuración. Mitigación: adaptar con `@WithMockUser` para tests de fondos, verificar que endpoints de auth permanecen públicos.
- `jwt.secret` en `application.properties` es para desarrollo. En producción debe externalizarse a variable de entorno o vault de secretos. Mitigación: usar valor seguro de 256+ bits y documentar.
- Reemplazo de `spring-security-crypto` por `spring-boot-starter-security` modifica transitividades. Mitigación: verificar con `./gradlew dependencies` tras el cambio.

**Patrones y convenciones del equipo:**
- Records para Comandos/Consultas/Resultados (DTOs en capa `application/`, NO en `api/dto/`)
- Inyección por constructor, nunca `@Autowired` en campo
- `ExcepcionDominio` para reglas de negocio (→422), excepciones específicas para HTTP diferenciados
- Puertos de dominio para dependencias externas (`PuertoRepositorio*`, `PuertoHashContrasena`)
- Bean Validation con mensajes en español en records
- Mediador auto-registra handlers por tipo — solo crear `@Component`
- `OncePerRequestFilter` para filtros transversales de seguridad
- `@Configuration` con `SecurityFilterChain` bean para configuración declarativa de Spring Security

**Dependencias nuevas a instalar:**
- `org.springframework.boot:spring-boot-starter-security` (reemplaza `spring-security-crypto`)
- `io.jsonwebtoken:jjwt-api:0.12.6`
- `io.jsonwebtoken:jjwt-impl:0.12.6` (runtimeOnly)
- `io.jsonwebtoken:jjwt-jackson:0.12.6` (runtimeOnly)
- `org.springframework.security:spring-security-test` (testImplementation)
