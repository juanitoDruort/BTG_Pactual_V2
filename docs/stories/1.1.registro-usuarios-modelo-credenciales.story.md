# Historia #1.1: Definición del proceso de registro de usuarios y modelo de credenciales

**Estado:** Lista para Revisión

---

## Historia de Usuario

**Como** administrador de la plataforma BTG Pactual,  
**Quiero** contar con una definición conceptual completa del proceso de auto-registro de clientes y el modelo de credenciales extendido,  
**Para** que el equipo de desarrollo tenga una especificación clara y sin ambigüedades que guíe la implementación del registro de usuarios en la plataforma de gestión de fondos de inversión.

## Descripción

Esta historia define conceptualmente el flujo de auto-registro de clientes en la plataforma de gestión de fondos de inversión de BTG Pactual. El sistema actualmente cuenta con una entidad `Cliente` (id, nombre, saldo) sin ningún mecanismo de seguridad. Se requiere extender el modelo de datos para incluir credenciales (email, teléfono, documento de identidad, contraseña) y definir las reglas de validación, política de contraseñas y manejo de excepciones del proceso de registro. El endpoint de registro es uno de los dos únicos puntos de acceso público (junto con login) de toda la plataforma.

---

## Criterios de Aceptación

### Escenario 1: Auto-registro exitoso de un cliente

- **Dado** que un usuario no registrado accede al endpoint público de registro
- **Cuando** envía sus datos completos (email, teléfono, documento de identidad, nombre y contraseña) y todos cumplen las validaciones definidas
- **Entonces** se crea el usuario con rol "Cliente" automáticamente, saldo inicial de COP $500.000 o mayor (definido en el registro), contraseña almacenada de forma segura, y se retorna una confirmación exitosa con los datos del usuario (sin exponer la contraseña)

### Escenario 2: Registro con contraseña inválida

- **Dado** que un usuario no registrado envía una solicitud de registro
- **Cuando** la contraseña proporcionada no cumple la política (mínimo 5 caracteres combinando números, letras y caracteres especiales)
- **Entonces** se rechaza el registro con un mensaje de error claro indicando los requisitos de la contraseña y código HTTP 400 (Bad Request)

### Escenario 3: Registro con email duplicado

- **Dado** que un usuario intenta registrarse
- **Cuando** el email proporcionado ya está registrado en el sistema
- **Entonces** se rechaza el registro con un mensaje de error que indique que el email ya está en uso, sin revelar información adicional del usuario existente, con código HTTP 409 (Conflict)

### Escenario 4: Registro con datos obligatorios faltantes

- **Dado** que un usuario envía una solicitud de registro
- **Cuando** falta alguno de los campos obligatorios (email, teléfono, documento de identidad, nombre o contraseña)
- **Entonces** se rechaza el registro con un mensaje de error que liste los campos faltantes, con código HTTP 400 (Bad Request)

### Escenario 5: Registro con formato de datos inválido

- **Dado** que un usuario envía una solicitud de registro con todos los campos
- **Cuando** algún campo tiene formato inválido (email sin formato correcto, teléfono con caracteres no numéricos, documento vacío, etc.)
- **Entonces** se rechaza el registro con un mensaje de error específico por cada campo inválido, con código HTTP 400 (Bad Request)

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Cualquier persona no registrada que desea convertirse en cliente de la plataforma
- **Permisos requeridos:** Ninguno — el registro es un endpoint público (no requiere autenticación)
- **Valor de negocio:** Permitir que los clientes se registren de manera autónoma sin necesidad de contactar un asesor, alineado con la visión de autogestión de fondos de inversión de BTG Pactual

### Reglas de Negocio

- El registro es por auto-registro (no requiere intervención de un administrador)
- Datos obligatorios: email, teléfono, documento de identidad, nombre y contraseña
- Política de contraseñas: mínimo 5 caracteres combinando números, letras y caracteres especiales
- El email es único por usuario (no se permiten duplicados)
- Al registrarse, el usuario recibe automáticamente el rol "Cliente"
- El saldo inicial del cliente es de COP $500.000 o mayor (el cliente puede definir su saldo inicial al registrarse, con un mínimo de COP $500.000)
- La contraseña debe almacenarse de forma segura (ver HU 1.4 para estrategia de encriptación)
- El endpoint de registro es público — uno de los dos únicos endpoints sin autenticación (junto con login)
- Todo el proceso debe incluir manejo adecuado de excepciones con mensajes claros

### Modelo de Datos Extendido del Cliente

| Campo | Descripción | Obligatorio |
|---|---|---|
| id | Identificador único del cliente (generado por el sistema) | Sí (auto) |
| nombre | Nombre completo del cliente | Sí |
| email | Correo electrónico (único, usado para login) | Sí |
| telefono | Número de teléfono del cliente | Sí |
| documentoIdentidad | Documento de identidad del cliente | Sí |
| contraseña | Contraseña hasheada del usuario | Sí |
| rol | Rol del usuario (CLIENTE o ADMINISTRADOR) | Sí (auto: CLIENTE) |
| saldo | Saldo disponible del cliente (mínimo COP $500.000) | Sí (>= COP $500.000) |
| bloqueado | Indicador de cuenta bloqueada | Sí (auto: false) |
| intentosFallidos | Contador de intentos fallidos de login | Sí (auto: 0) |

### Relación con Dominio Existente

- La entidad `Cliente` actual (id, nombre, saldo) se extiende con los campos de credenciales y seguridad
- Dentro de la arquitectura hexagonal, el modelo extendido vive en la capa de dominio (`domain/model/`)
- Los puertos de repositorio (`PuertoRepositorioCliente`) deben soportar las nuevas operaciones de búsqueda por email y verificación de existencia

### Sistemas Externos

No aplica — el registro es un proceso completamente interno de la plataforma.

## Análisis Arquitectónico (Arquitecto)

### Decisiones de Diseño

**Patrón Arquitectónico:** Arquitectura Hexagonal + CQRS + Mediador (extensión coherente del patrón existente)

**Justificación:** El sistema ya implementa exitosamente este patrón para suscripción y consulta de fondos (`FondoComando → FondoManejador` vía `Mediador`). El flujo de registro encaja naturalmente como un nuevo Command (`RegistroComando → RegistroManejador`) que se registra automáticamente en el Mediador sin modificar controladores existentes ni alterar flujos actuales. Es la opción de menor fricción y mayor coherencia con la base de código verificada. Los DTOs de entrada/salida se ubican en la capa de aplicación (no en API) para que los manejadores CQRS tengan acceso directo a los objetos sobre los cuales toman decisiones.

**Componentes Afectados:**

- **Rol.java (Nuevo):** Enum en `domain/model/` que define los roles del sistema (CLIENTE, ADMINISTRADOR). Base para autorización futura (HU 1.3).

- **Cliente.java (Modificación — Mayor):** Extender modelo de dominio de 3 campos (id, nombre, saldo) a modelo completo con credenciales y seguridad. Agregar: email, telefono, documentoIdentidad, contrasenaHash, rol (enum Rol), cuentaBloqueada, intentosFallidosLogin, fechaRegistro. Agregar método `abonarSaldo(BigDecimal)` para HU 2.2 futura. Constructor extendido; mantener compatibilidad con datos precargados.
  - Nivel de cambio: Mayor

- **PuertoRepositorioCliente.java (Modificación — Menor):** Agregar dos métodos al puerto de salida: `Optional<Cliente> buscarPorEmail(String email)` y `boolean existePorEmail(String email)`. Necesarios para validar unicidad de email en registro.
  - Nivel de cambio: Menor

- **AdaptadorClienteEnMemoria.java (Modificación — Mayor):** Implementar `buscarPorEmail()` y `existePorEmail()`. Actualizar datos iniciales precargados para incluir los nuevos campos del modelo extendido (email, telefono, documentoIdentidad, contrasenaHash, rol, etc.).
  - Nivel de cambio: Mayor

- **RegistroComando.java (Nuevo):** Record en `application/registro/command/` con Bean Validation (@NotBlank, @Email, @Pattern, @Min). Cumple el rol de DTO de entrada directamente — el controlador lo recibe como @RequestBody @Valid y lo envía al Mediador.
  - Especificaciones: nombre, email, telefono, documentoIdentidad, contrasena, saldoInicial

- **RegistroResultado.java (Nuevo):** Record en `application/registro/command/` con: clienteId, nombre, email, rol. Sin exponer contraseña ni datos sensibles. Sirve como respuesta directa del controlador.

- **RegistroManejador.java (Nuevo):** @Component en `application/registro/command/` que implementa `Manejador<RegistroComando, RegistroResultado>`. Orquesta: validar unicidad email → validar política contraseña → hash contraseña con BCrypt → crear Cliente con rol CLIENTE y defaults de seguridad → guardar → retornar resultado.

- **ControladorAutenticacion.java (Nuevo):** Controlador REST en `api/controller/`. Endpoint `POST /api/auth/registro` — recibe `RegistroComando` como @RequestBody @Valid, envía al Mediador, retorna `RegistroResultado` con HTTP 201. @RestController, @RequestMapping("/api/auth"), @Tag OpenAPI.

- **ManejadorExcepcionesGlobal.java (Modificación — Menor):** Agregar handler para respuesta HTTP 409 (Conflict) cuando email ya existe. Captura una nueva excepción de dominio específica o una ExcepcionDominio con código de conflicto.
  - Nivel de cambio: Menor

- **build.gradle (Modificación — Menor):** Agregar dependencia `org.springframework.security:spring-security-crypto` para BCrypt (PasswordEncoder). NO agregar `spring-boot-starter-security` completo — la configuración de filtros de seguridad (SecurityFilterChain, JWT) pertenece a HU 1.2/1.3.
  - Nivel de cambio: Menor

**Hitos de Implementación:**

1. **Enum Rol** - Crear `domain/model/Rol.java` (CLIENTE, ADMINISTRADOR)
   - Dependencias: Ninguna

   - Dependencias: Rol

   - Dependencias: Cliente extendido

   - Dependencias: Puerto actualizado, Cliente extendido

### Validación de Impacto

Se verificó el código fuente real de todos los componentes involucrados:

- **Cliente.java:** Actualmente tiene solo 3 campos (id, nombre, saldo) y un constructor simple. NO soporta credenciales — requiere extensión mayor. El método `descontarSaldo()` se mantiene intacto.
- **PuertoRepositorioCliente:** Solo tiene `buscarPorId()` y `guardar()`. NO tiene búsqueda por email — requiere adición de métodos.
- **AdaptadorClienteEnMemoria:** Precarga 2 clientes con constructor de 3 params (`"cliente-1", "Juan Rodriguez", 500000`). Debe adaptarse al constructor extendido y agregar búsqueda por stream sobre email.
- **Mediador + Manejador:** Patrón genérico funcional — solo requiere registrar el nuevo `RegistroManejador` como @Component. Sin modificaciones al mecanismo.
- **ManejadorExcepcionesGlobal:** Maneja `ExcepcionDominio` → 422 y `MethodArgumentNotValidException` → 400. Falta handler para 409.
- **build.gradle:** No tiene dependencia de seguridad — requiere agregar `spring-security-crypto`.
- **Tests existentes (ControladorFondoE2ETest):** Usan `AdaptadorClienteEnMemoria` con datos precargados de 3 campos. Requerirán ajuste al cambiar el constructor de Cliente.

### Notas Técnicas (Si aplica)

- **Decisión de DTOs en capa de aplicación:** Por decisión del arquitecto, los DTOs de registro (`RegistroComando` como entrada, `RegistroResultado` como salida) se ubican en la capa de aplicación (`application/registro/command/`), no en `api/dto/`. Esto permite que el manejador CQRS tenga acceso directo a los objetos sobre los que toma decisiones, manteniendo la capa de aplicación autosuficiente.
- **Separación de spring-security-crypto vs starter-security:** Se agrega únicamente la librería de hashing (BCrypt) sin activar el autoconfigurador completo de Spring Security. El SecurityFilterChain y JWT se configurarán en HU 1.2. Si hay incompatibilidad, alternativa: `at.favre.lib:bcrypt`.
- **Impacto en tests existentes:** La modificación de `Cliente.java` romperá `ControladorFondoE2ETest` y datos precargados en `AdaptadorClienteEnMemoria`. Se debe actualizar el constructor y los datos de prueba simultáneamente.
- **Validación de contraseña:** La política (mín 5 chars, letras + números + especiales) se implementa con @Pattern en `RegistroComando` más validación explícita en `RegistroManejador` como doble capa de seguridad.

## Refinamiento Técnico (Developer)

### Consideraciones Generales

**Basado en análisis arquitectónico:**
Sección "Análisis Arquitectónico (Arquitecto)" de esta misma historia — 8 hitos de implementación, 10 componentes afectados. Extensión mayor del modelo `Cliente`, nuevo flujo CQRS de registro, nueva dependencia BCrypt, nuevo controlador REST público.

**Nivel de complejidad:**
MEDIA - Nuevo flujo completo CQRS (RegistroComando→RegistroManejador→RegistroResultado) con extensión mayor del modelo de dominio (3→10+ campos), creación de 2 nuevas excepciones y puertos, nueva dependencia externa (spring-security-crypto), y breaking change en constructor de `Cliente` que impacta adaptador en memoria y tests existentes. No hay migración de datos (in-memory), pero sí requiere coordinación entre 4 capas (domain, application, infrastructure, api).

**Riesgos técnicos conocidos:**
- Cambio de constructor de `Cliente.java` rompe `AdaptadorClienteEnMemoria.cargarDatosIniciales()` y tests existentes (`ControladorFondoE2ETest`, `BtgPactualV2ApplicationTests`). Mitigación: actualizar adaptador y tests en la misma fase.
- `spring-security-crypto` podría arrastrar transitividades de Spring Security que activen autoconfiguración. Mitigación: verificar con `./gradlew dependencies` tras agregar; fallback a `at.favre.lib:bcrypt:0.10.2`.
- `@Pattern` para validación de contraseña en `RegistroComando` podría no cubrir todos los edge cases de la política (5+ chars, letras + números + especiales). Mitigación: doble validación en handler con lógica explícita.

**Patrones y convenciones del equipo:**
- Records para Comandos/Consultas/Resultados (DTOs en capa application, NO en api/dto/)
- Inyección por constructor, nunca `@Autowired` en campo
- BigDecimal para valores monetarios
- Modelos de dominio con lógica encapsulada (no anémicos)
- `ExcepcionDominio` para reglas de negocio (→422), `ExcepcionConflicto` para conflictos de unicidad (→409)
- Puertos de dominio para dependencias externas (`PuertoRepositorio*`, `PuertoHashContrasena`)
- Bean Validation con mensajes en español directamente en records
- Mediador auto-registra handlers por tipo — solo crear `@Component`

**Dependencias nuevas a instalar:**
`org.springframework.security:spring-security-crypto` — módulo aislado para BCrypt. NO incluir `spring-boot-starter-security` (activa SecurityFilterChain, pertenece a HU 1.2).

**Estrategia de testing:**
JUnit 5 + MockMvc + Mockito | Tests E2E (ControladorAutenticacion: 5 escenarios AC1-AC5) + Tests unitarios (RegistroManejador: 4 casos) | Cobertura: AC1-AC5 con tests directos | Builders: RegistroComandoBuilder (crear nuevo), ClienteBuilder (crear nuevo)

**Implementaciones similares analizadas:**
