# Historia #3: Diseño del modelo de datos NoSQL en Amazon DynamoDB

**Estado:** Lista para Revisión (Developer)  
**Tipo:** Historia Técnica (Implementación)

---

## Historia de Usuario

**Como** equipo de desarrollo de la plataforma BTG Pactual,  
**Quiero** contar con un diseño documentado del modelo de datos NoSQL en Amazon DynamoDB que soporte todas las operaciones del sistema,  
**Para** tener una especificación clara de tablas, estructura de ítems, partition/sort keys, Global Secondary Indexes (GSI), estrategia de denormalización y datos semilla que guíe la implementación de la capa de persistencia.

## Descripción

El sistema actualmente opera con persistencia en memoria (ConcurrentHashMap) a través de adaptadores in-memory en la capa de infraestructura. Esta historia técnica define el diseño completo del modelo de datos para Amazon DynamoDB, incluyendo la estructura de cada tabla, la estrategia de denormalización, las partition/sort keys optimizadas para los patrones de acceso identificados, los Global Secondary Indexes (GSI) necesarios, y los datos semilla para inicialización. El modelo debe contemplar las 3 entidades de dominio actuales (Cliente, Fondo, Suscripción) con la extensión del modelo de Cliente definida en HU 1.1 (campos de seguridad). El entorno de desarrollo utiliza DynamoDB Local en Docker. Este diseño es el artefacto de referencia que el equipo de desarrollo usará para implementar los nuevos adaptadores de persistencia.

## Contexto y Referencias

**Arquitectura:** Arquitectura Hexagonal + CQRS + Mediador según [ARQUITECTURA.md](../../ARQUITECTURA.md). El modelo de datos DynamoDB se alinea con los puertos de repositorio definidos en la capa de dominio (`PuertoRepositorioCliente`, `PuertoRepositorioFondo`, `PuertoRepositorioSuscripcion`). Los adaptadores in-memory actuales no se modifican — este diseño es la base para futuros adaptadores de DynamoDB en `infrastructure/adapter/out/persistence/`.

**Historias relacionadas:**
- HU 1.1 (Registro — define extensión del modelo de Cliente que se incluye en este diseño)
- HU 2.1-2.4 (Operaciones de suscripción — definen los patrones de acceso a soportar)
- HU 1.4 (Encriptación — la contraseña se almacena como hash, no en texto plano)

**Lecciones aprendidas:**
- Los puertos de repositorio actuales definen el contrato mínimo; el diseño NoSQL debe soportar al menos esas operaciones
- `PuertoRepositorioSuscripcion` carece de `buscarPorClienteId()` — el diseño debe contemplar este patrón (requerido por HU 2.3)
- La verificación de duplicados actual filtra por estado ACTIVO — esto debe reflejarse en el diseño de partition keys y FilterExpressions
- DynamoDB no tiene unique key constraints nativos fuera de PK/SK — la unicidad de email se garantiza por lógica aplicativa
- Los GSI (Global Secondary Indexes) reemplazan la política de indexación de Cosmos DB para accesos secundarios

### Tabla `clientes`

**Partition Key:** `id` (String)  
**Sort Key:** — (no aplica)

**GSI `email-index`:**
- Partition Key: `email` (String)
- Proyección: ALL (todos los atributos)

**Justificación:** GetItem por `id` es la operación más frecuente (cada suscripción/cancelación consulta al cliente). Query por email (login/registro) se resuelve eficientemente con el GSI `email-index`. No se requiere sort key porque cada cliente es un ítem único identificado por su ID.

**Estructura del ítem:**

```json
{
  "id":                        { "S": "cliente-1" },
  "nombre":                    { "S": "Juan Rodriguez" },
  "saldo":                     { "N": "500000" },
  "email":                     { "S": "juan.rodriguez@email.com" },
  "telefono":                  { "S": "+573001234567" },
  "documentoIdentidad":        { "S": "1234567890" },
  "contrasenaHash":              { "S": "$2a$10$...(hash BCrypt)" },
  "rol":                       { "S": "CLIENTE" },
  "bloqueado":                 { "BOOL": false },
  "intentosFallidos":          { "N": "0" },
  "preferenciaNotificacion":   { "S": "EMAIL" },
  "fechaRegistro":             { "S": "2026-01-01T00:00:00" }
}
```

**Unicidad de email:** DynamoDB no soporta unique key constraints fuera de la primary key. La unicidad se garantiza mediante `existePorEmail()` (Query al GSI) antes de `guardar()`, igual que el patrón actual en adaptadores in-memory.

**Operaciones:**

| Operación | API DynamoDB | Eficiencia |
|---|---|---|
| `buscarPorId("cliente-1")` | `GetItem(TableName="clientes", Key={id: "cliente-1"})` | 0.5 RCU (eventually consistent) |
| `guardar(cliente)` | `PutItem(TableName="clientes", Item=doc)` | 1 WCU |
| `existePorEmail(email)` | `Query(IndexName="email-index", KeyConditionExpression="email = :e", Limit=1)` | 0.5 RCU (GSI) |
| `buscarPorEmail(email)` | `Query(IndexName="email-index", KeyConditionExpression="email = :e")` | 0.5 RCU (GSI) |

### Tabla `suscripciones`

**Partition Key:** `clienteId` (String)  
**Sort Key:** `id` (String) — UUID de la suscripción  
**GSI `id-index`:**
- Partition Key: `id` (String)
- Proyección: ALL (todos los atributos)
- Justificación: Soporta `buscarPorId(id)` sin conocer `clienteId`

**Justificación:** Patrón dominante es listar suscripciones activas por cliente (Query single-partition con FilterExpression). Verificación de duplicados también single-partition. GetItem por ID de suscripción usa PK (`clienteId` del JWT) + SK (`id`). Todos los patrones parten de `clienteId`, sin necesidad de GSI adicional.

**Estructura del ítem (con denormalización):**

```json
{
  "clienteId":        { "S": "cliente-1" },
  "id":               { "S": "uuid-suscripcion-1" },
  "fondoId":          { "S": "fondo-1" },
  "nombreFondo":      { "S": "FPV_BTG_PACTUAL_RECAUDADORA" },
  "monto":            { "N": "75000" },
  "estado":           { "S": "ACTIVO" },
  "fechaSuscripcion": { "S": "2026-03-12T10:30:00" }
}
```

**Denormalización `nombreFondo`:**
- HU 2.3 necesita nombre del fondo al listar vigentes. Sin embebido = N queries a `fondos`.
- Fondos son catálogo inmutable → sin riesgo de inconsistencia.
- `fondoId` se mantiene como referencia canónica.

**Operaciones:**

| Operación | API DynamoDB | Eficiencia |
|---|---|---|
| `guardar(suscripcion)` | `PutItem(TableName="suscripciones", Item=doc)` | 1 WCU (single-partition) |
| `existePorClienteIdYFondoId(c, f)` | `Query(PK=clienteId, FilterExpression="fondoId = :fid AND estado = :activo")` | Single-partition query |
| `buscarPorId(id)` | `GetItem(Key={clienteId: cid, id: id})` | 0.5 RCU (point read, clienteId del JWT) |
| `buscarPorClienteIdYEstado(c, ACTIVO)` | `Query(KeyConditionExpression="clienteId = :cid", FilterExpression="estado = :activo")` | Single-partition query |

### Creación de Tablas (CLI de referencia)

```bash
# Tabla clientes
aws dynamodb create-table \
  --table-name clientes \
  --attribute-definitions \
    AttributeName=id,AttributeType=S \
    AttributeName=email,AttributeType=S \
  --key-schema AttributeName=id,KeyType=HASH \
  --global-secondary-indexes \
    'IndexName=email-index,KeySchema=[{AttributeName=email,KeyType=HASH}],Projection={ProjectionType=ALL}' \
  --billing-mode PAY_PER_REQUEST \
  --endpoint-url http://localhost:8000

# Tabla fondos
aws dynamodb create-table \
  --table-name fondos \
  --attribute-definitions AttributeName=id,AttributeType=S \
  --key-schema AttributeName=id,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --endpoint-url http://localhost:8000

# Tabla suscripciones
aws dynamodb create-table \
  --table-name suscripciones \
  --attribute-definitions \
    AttributeName=clienteId,AttributeType=S \
    AttributeName=id,AttributeType=S \
  --key-schema \
    AttributeName=clienteId,KeyType=HASH \
    AttributeName=id,KeyType=RANGE \
  --global-secondary-indexes \
    'IndexName=id-index,KeySchema=[{AttributeName=id,KeyType=HASH}],Projection={ProjectionType=ALL}' \
  --billing-mode PAY_PER_REQUEST \
  --endpoint-url http://localhost:8000
```

### Hitos de Implementación

| Hito | Entregable | Verificación |
|---|---|---|
| **H1** — Tabla `clientes` | Estructura PK `id`, GSI `email-index` | Cubre operaciones #1-#4 |
| **H2** — Tabla `fondos` | Estructura PK `id` | Cubre operación #5 |
| **H3** — Tabla `suscripciones` | Estructura PK `clienteId` + SK `id`, denormalización `nombreFondo` | Cubre operaciones #6-#9 |
| **H4** — Datos semilla | 2 clientes (campos extendidos), 5 fondos — script PutItem | Consistentes con modelo |
| **H5** — Validación de patrones | 9 operaciones con API DynamoDB + estimación RCU/WCU | Cada patrón documentado |

### Decisiones Arquitectónicas

1. **PK `clienteId` + SK `id` en suscripciones** — Permite Query single-partition para listar activas y GetItem eficiente por ID (clienteId disponible desde JWT).
2. **GSI `email-index` en clientes** — Único GSI necesario en todo el diseño. Soporta login y verificación de email duplicado.
3. **Sin GSI en suscripciones** — Todos los patrones de acceso parten de `clienteId`. FilterExpression en `estado`/`fondoId` es suficiente dado el volumen esperado por partición.
4. **Denormalización `nombreFondo`** — Evita N lookups al listar vigentes. Catálogo de fondos inmutable = sin riesgo de inconsistencia.
5. **On-Demand capacity (PAY_PER_REQUEST)** — Óptimo para desarrollo/test. Migrable a Provisioned con Auto Scaling en producción.
6. **3 tablas separadas** — Entidades con ciclos de vida distintos. Single-table design agrega complejidad innecesaria para los patrones de acceso actuales (9 operaciones simples).
7. **Unicidad de email por lógica aplicativa** — DynamoDB no tiene unique key constraints fuera de PK/SK. Se mantiene `existePorEmail()` + validación antes de `guardar()`.
8. **`saldo` como número en ítem** — HU 1.4 (AES-256-GCM) se aplica en el adaptador de persistencia (encripta al guardar, desencripta al leer).
9. **AWS SDK v2 Enhanced Client** — Mapeo directo entre records/POJOs Java y DynamoDB ítems. Reduce boilerplate vs. SDK clásico.
10. **DynamoDB Local para desarrollo** — Docker image `amazon/dynamodb-local`. Sin necesidad de cuenta AWS ni credenciales reales.

**Riesgos técnicos conocidos:**
- `PuertoRepositorioSuscripcion.buscarPorId(String id)` solo recibe `id`, pero DynamoDB necesita PK `clienteId` + SK `id`. Solución: GSI `id-index` en tabla suscripciones para soportar GetItem por `id` solo.
- `AdaptadorClienteEnMemoria` encripta/desencripta saldo con `PuertoEncriptacion`. Los adaptadores DynamoDB deben replicar este patrón.
- `Cliente.java` tiene `fechaRegistro` (LocalDateTime) que no estaba en el diseño original. Corregido: se agrega al esquema.
- Naming: campo en código es `contrasenaHash`, en diseño era `contrasena`. Corregido: usar `contrasenaHash` en DynamoDB.
- `preferenciaNotificacion` aún no existe en `Cliente.java` actual — se incluye en diseño como campo proyectado de HU futura.

**Patrones y convenciones del equipo:**
- Adaptadores con prefijo `Adaptador*DynamoDb` (ej: `AdaptadorClienteDynamoDb`) según coding-standards.md
- Inyección por constructor, nunca `@Autowired` en campo
- BigDecimal para valores monetarios
- Records para comandos/consultas/resultados
- `@Profile("dynamodb")` en adaptadores DynamoDB, `@Profile("!dynamodb")` en adaptadores in-memory
- AWS SDK v2 Enhanced Client con anotaciones `@DynamoDbBean`/`@DynamoDbPartitionKey`/`@DynamoDbSortKey`

**Dependencias nuevas a instalar:**
- `software.amazon.awssdk:bom:2.29.51` (platform)
