# Historia #2.3: Consulta de suscripciones vigentes del cliente

**Estado:** Analizado (Arquitecto)

---

## Historia de Usuario

**Como** cliente de la plataforma BTG Pactual,  
**Quiero** consultar el listado de mis suscripciones vigentes a fondos de inversión,  
**Para** conocer mi portafolio activo, ver cuánto tengo invertido en cada fondo y poder decidir cuáles cancelar si lo deseo.

## Descripción

El cliente puede consultar un listado con todas sus suscripciones que se encuentran en estado ACTIVO. Este listado le permite visualizar su portafolio de inversiones vigente, incluyendo el nombre del fondo, monto invertido, fecha de suscripción e identificador de cada suscripción. Esta consulta es esencial para que el cliente identifique cuál suscripción cancelar (por ID) en la funcionalidad de cancelación (HU 2.2).

---

## Criterios de Aceptación

### Escenario 1: Cliente con suscripciones activas

- **Dado** que un cliente tiene una o más suscripciones con estado ACTIVO
- **Cuando** consulta su listado de suscripciones vigentes
- **Entonces** se retorna únicamente las suscripciones con estado ACTIVO de ese cliente, excluyendo las CANCELADAS

### Escenario 2: Cliente sin suscripciones vigentes

- **Dado** que un cliente no tiene suscripciones con estado ACTIVO (nunca se suscribió o canceló todas)
- **Cuando** consulta su listado de suscripciones vigentes
- **Entonces** se retorna una lista vacía con código HTTP 200

### Escenario 3: Datos incluidos en cada suscripción del listado

- **Dado** que un cliente tiene suscripciones activas
- **Cuando** recibe el listado
- **Entonces** cada suscripción incluye: ID de suscripción, ID del fondo, nombre del fondo, monto invertido, fecha de suscripción y estado

### Escenario 4: Aislamiento de datos entre clientes

- **Dado** que existen suscripciones de múltiples clientes en el sistema
- **Cuando** un cliente consulta su listado
- **Entonces** solo se retornan las suscripciones que pertenecen a ese cliente específico (aislamiento por userId del token JWT, según HU 1.3)

---

## Información Recopilada

### Usuario y Contexto

- **Tipo de usuario:** Cliente registrado y autenticado en la plataforma (ver HU 1.1 y HU 1.2)
- **Permisos requeridos:** Rol CLIENTE — solo puede consultar sus propias suscripciones (ver HU 1.3, aislamiento de datos)
- **Valor de negocio:** Proporcionar transparencia al cliente sobre su portafolio activo de fondos de inversión, permitiéndole tomar decisiones informadas sobre cancelaciones o nuevas suscripciones

### Reglas de Negocio

- Solo retorna suscripciones con estado ACTIVO del cliente autenticado
- Si no tiene suscripciones activas, se retorna lista vacía (no error)
- El aislamiento de datos garantiza que un cliente nunca ve suscripciones de otro cliente
- El listado sirve como insumo para seleccionar cuál suscripción cancelar (HU 2.2)

### Datos Incluidos por Suscripción

| Campo | Descripción |
|---|---|
| suscripcionId | Identificador único de la suscripción |
| fondoId | Identificador del fondo |
| nombreFondo | Nombre del fondo de inversión |
| monto | Monto invertido en el fondo |
| fechaSuscripcion | Fecha y hora de la suscripción |
| estado | Estado de la suscripción (siempre ACTIVO en este listado) |

### Sistemas Externos

No aplica — es una consulta de solo lectura sobre datos internos.

## Contexto y Referencias

**Arquitectura:** Arquitectura Hexagonal + CQRS + Mediador según [ARQUITECTURA.md](../../ARQUITECTURA.md). Esta funcionalidad es una Query pura en el patrón CQRS: SuscripcionesVigentesConsulta + SuscripcionesVigentesManejador → PuertoRepositorioSuscripcion.buscarPorClienteIdYEstado().

**Historias relacionadas:** HU 2.1 (crea las suscripciones a listar), HU 2.2 (usa el listado para identificar cuál cancelar)

**Historias hermanas (Slicing):** HU 2.1 (Suscripción), HU 2.2 (Cancelación), HU 2.3 (esta), HU 2.4 (Notificación)

**Lecciones aprendidas:**
- PuertoRepositorioSuscripcion actualmente solo tiene guardar() y existePorClienteIdYFondoId() — necesita extensión con buscarPorClienteIdYEstado()
- El adaptador en memoria necesita filtrado por clienteId y estado

## Análisis Arquitectónico

**Fecha de análisis:** 2026-03-12  
**Enfoque:** Exploratorio (B)  
**Arquitecto:** Agente Arquitecto Método Ceiba

### Hallazgos del Análisis

| Aspecto | Hallazgo |
|---|---|
| Patrón Query existente | `FondoConsulta → FondoManejador(query) → IServicioFondo.obtener()` — HU 2.3 replica este patrón pero retorna **lista** en vez de entidad única |
| Mediador | Soporta `<T, R>` genérico — `R` puede ser `List<X>` sin modificaciones |
| `Suscripcion.java` | Solo tiene `fondoId` (String), **NO tiene `nombreFondo`** — AC3 exige retornar nombre del fondo. Se resuelve en el Handler vía `PuertoRepositorioFondo` |
| `PuertoRepositorioSuscripcion` | No tiene búsqueda por clienteId + estado — necesita `buscarPorClienteIdYEstado()` |
| `AdaptadorSuscripcionEnMemoria` | Patrón de filtrado por stream ya existente en `existePorClienteIdYFondoId()` — replicable |
| DTOs existentes | `FondoResultado(command)` y `RespuestaSuscripcion` no tienen `nombreFondo` — no reutilizables |

### Hallazgo Crítico: Enriquecimiento con nombre del fondo

La `Suscripcion` del dominio solo almacena `fondoId`. El AC3 exige retornar `nombreFondo`. Opciones evaluadas:

| Opción | Descripción | Decisión |
|---|---|---|
| A | Guardar `nombreFondo` en Suscripcion | ❌ Desnormalización innecesaria en dominio |
| B | Resolver `nombreFondo` en el Handler (capa aplicación) vía `PuertoRepositorioFondo` | ✅ **Elegida** — query handler tiene acceso al puerto, resuelve al construir DTO |
| C | Resolver en el Controlador | ❌ Viola arquitectura hexagonal |

### Hitos de Implementación

| Hito | Componentes | Verificación |
|---|---|---|
| **H1** — Puerto y adaptador de búsqueda | `PuertoRepositorioSuscripcion.buscarPorClienteIdYEstado()`, `AdaptadorSuscripcionEnMemoria.buscarPorClienteIdYEstado()` | Test unitario: filtra por clienteId y estado, retorna lista vacía si no hay |
| **H2** — Query handler con enriquecimiento | `SuscripcionesVigentesConsulta`, `SuscripcionVigenteResultado`, `SuscripcionesVigentesManejador` | Test unitario: resuelve nombreFondo, filtra solo ACTIVAS, aislamiento por clienteId |
| **H3** — Endpoint REST | `ControladorSuscripcion +GET /api/suscripciones/vigentes` | Test E2E: lista con vigentes, lista vacía HTTP 200, aislamiento entre clientes |

### Decisiones Arquitectónicas

1. **Handler inyecta puertos directamente (no pasa por servicio de dominio)** — Query pura sin lógica de negocio. Sigue principio CQRS: queries no necesitan los mismos servicios que commands.
2. **`SuscripcionVigenteResultado` ES el DTO de respuesta** — DTOs en capa aplicación (decisión HU 1.1). No se crea nada en `api/dto/`. El controlador retorna directamente el Resultado de aplicación.
3. **Paquete `application/suscripcion/query/`** — Separado de `application/cancelacion/command/` (HU 2.2). Organización por feature + CQRS.
4. **Lista vacía = HTTP 200 (no 404)** — AC2: ausencia de vigentes no es error.
